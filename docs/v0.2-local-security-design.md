# WorkTrace Local Security and Backup Contract

This document describes the current pre-release local security and encrypted
backup design. Earlier phase descriptions and compatibility formats are retired.

## Security boundary

WorkTrace is local-only: there is no server, no account, no cloud sync, no AI
processing and no telemetry upload. It requires no administrator privilege or
network connection. It does not capture screenshots, screen recordings, keyboard
input, camera or microphone data.

The local application key is protected by Windows DPAPI. Sensitive local fields
use authenticated encryption (AEAD) with AES-GCM. The current implementation does
not use SQLCipher and does not claim full-database transparent encryption.
Argon2id remains the password-hardening design for user-secret protection where
that capability is used; encrypted portable backup derivation uses the explicitly
versioned scrypt parameters recorded in the backup manifest.

A `.wtbackup` file is created only by an explicit user action. The passphrase is
chosen by the user and is not recoverable by WorkTrace. Logs and UI errors never
contain passphrases, decrypted payloads, full ciphertext, local backup paths,
window titles, file paths, notes or clipboard text.

The encrypted container uses an AEAD envelope. Wrong passphrase, corrupted
ciphertext, invalid AAD or malformed manifest fails without plaintext downgrade.
The manifest contains non-sensitive format and KDF metadata.

## Current-only payload

The only accepted business payload is:

```json
{
  "format": "worktrace-local-data",
  "version": 6,
  "created_at": "<ISO 8601 UTC>",
  "app_version": "<current app version>",
  "schema_version": "12",
  "schema_fingerprint": "<exact current fingerprint>",
  "tables": { "<exact current export table set>": [] }
}
```

Payload v6 requires schema v12 and the exact current schema fingerprint. The
application is not released, so old payloads and old schemas are rejected rather
than migrated. Export does not emit dual fields or compatibility metadata.

Derived and operational state is excluded, including folder file-index rows and
background-worker job/progress tables. Installation privacy consent and the
machine keyring are not business backup data. Clipboard events may be present in
the encrypted payload when the user has enabled that feature; the UI warns that
the backup may contain copied text.

## Export

`secure_backup_service.export_encrypted_backup()` owns the complete use case:

1. validate passphrase and target bounds;
2. acquire `RuntimeMaintenanceCoordinator.consistent_snapshot()`;
3. require Collector HELD before write-gate draining;
4. read the exact current schema fingerprint and export table set;
5. release maintenance according to the captured durable intent;
6. encrypt the payload; and
7. atomically replace the output file.

API and bridge layers do not build payloads and cannot provide an alternate
snapshot barrier.

## Replace import

Only replace import is supported. Merge import is not supported.

1. Bound and decrypt the input file before destructive work.
2. Parse and validate payload v6, schema v12, fingerprint, exact table set and
   row shapes.
3. Build a staging SQLite database from the current schema and validate its
   foreign keys and semantic invariants.
4. Acquire `RuntimeMaintenanceCoordinator.database_replacement()`.
5. Hold Collector, clear runtime snapshot, drain writers and enter exclusive
   ownership.
6. Replace business rows and publish the database-replacement epoch in the same
   transaction.
7. Release exclusive ownership, reset process-local Collector/adapter identities
   while still HELD, restore durable settings and release Collector.

Wrong passphrase, corrupted or unsupported payload, or staging validation failure
cannot modify the live database. A replacement transaction failure rolls back
business data and the replacement epoch. Unknown Collector command state fails
closed and does not fabricate import success.

## Maintenance dependency direction

```text
SecureBackupService -> RuntimeMaintenanceCoordinator -> CollectorControl
```

Collector never imports or queries backup service to discover maintenance state.
The old `secure_import_in_progress` lifecycle flag and reverse dependency are
removed. The UI may read the coordinator's narrow active status through
`backup_api`; that does not make backup service a Collector dependency.
Maintenance hold is not represented as user pause or as a session boundary.

## Privacy and restoration

Installation privacy acceptance is stored outside backup business data and is
preserved across export, import and clear-all. The coordinator restores the
captured durable user intent:

- pre-existing user pause remains paused;
- a running, authorized, non-user-paused Collector resumes;
- an unaccepted privacy notice remains stopped;
- a previously stopped runtime remains stopped.

Unknown command or failed release state commits the stable fail-closed safety
latch. Destructive maintenance remains blocked until explicit runtime recovery is
verified.

## Validation

Tests cover exact payload/schema versions, fingerprint, encrypted round trip,
old-version rejection, corrupted/wrong-passphrase failure, staging validation,
worker-progress exclusion, replacement generation, maintenance concurrency and
privacy restoration. Windows package smoke verifies the security dependencies
remain compatible with the single-file executable and installer.
