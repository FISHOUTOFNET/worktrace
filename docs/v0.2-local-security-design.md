# WorkTrace v0.2 Local Security Design

## Phase 1A Scope

This phase establishes the local security foundation for later v0.2 work. It adds:

- local security foundation;
- encrypted backup/import foundation;
- Windows current-user DPAPI keyring;
- AEAD envelope format;
- `.wtbackup` encrypted backup container foundation;
- no AI;
- no server;
- no payment, license, token, or subscription feature;
- no SQLCipher;
- no runtime DB field encryption yet.

Phase 1A does not change the existing runtime database schema, startup flow, collector behavior, UI behavior, statistics, export, or user-data write path.

## Route

The local security work is split into small phases:

1. Phase 1A: crypto foundation, DPAPI keyring, and `.wtbackup` format.
2. Phase 1B: encrypted backup/export/import integration.
3. Phase 1C: first batch high-sensitive field encryption.
4. Phase 1D: project, rule, folder-index encryption and blind indexes.

## SQLCipher Decision

SQLCipher is not adopted in Phase 1A.

The current Python, Windows, and PyInstaller distribution chain is uncertain for SQLCipher. Unofficial wheels are not suitable as a formal WorkTrace runtime dependency, and maintaining a custom SQLCipher wheel would add a high maintenance burden. A SQLCipher spike may remain useful for future investigation, but it is not part of this phase.

## Runtime Database Boundary

Phase 1A does not modify the existing runtime database. It does not add encrypted database columns, change existing business fields, require startup migration, change collection writes, or change how existing WorkTrace user data is stored during normal app operation.

The new security modules are independent foundations for future integration.

## Cryptography

The AEAD envelope uses AES-GCM with a fresh 96-bit random nonce for each encryption operation. Callers cannot provide a nonce. The envelope is versioned as `wtenc1:<base64url-json>` and contains only encryption metadata and ciphertext.

Backup passphrase derivation targets Argon2id as the preferred long-term design. Phase 1A uses scrypt as the packaging-conservative fallback because it is available through `cryptography`, keeps the dependency set small, and is more predictable for Windows/PyInstaller than adding Argon2 bindings at this stage. Backup manifests serialize the KDF algorithm, parameters, and unique random salt so the format can migrate later.

## Security Goals

- Backup files cannot be viewed as plaintext.
- The keyring never stores plaintext data keys.
- Encryption failure must not silently downgrade to plaintext.
- A wrong backup password must not decrypt.
- Nonces must not be reused by the encryption API.
- AAD mismatch must fail.
- Logs and errors must not include keys, passwords, plaintext sensitive content, or complete ciphertext.

## Not Protected

Phase 1A does not protect against:

- malware running as the same Windows user;
- runtime memory capture;
- plaintext Excel files that the user actively exports.

## Acceptance Criteria

Phase 1A is accepted when:

- `worktrace/security` contains independent crypto, KDF, key manager, and `.wtbackup` modules.
- AES-GCM round trips work and wrong keys, wrong AAD, corrupted ciphertext, and malformed envelopes fail.
- Re-encrypting the same plaintext produces different envelopes.
- The DPAPI keyring stores wrapped data keys at `%LOCALAPPDATA%\WorkTrace\security\keyring.json` and never stores plaintext data keys.
- Non-Windows and tests can use a fake key wrapper without real DPAPI.
- `.wtbackup` can encrypt/decrypt a payload with a passphrase, reject wrong passphrases, reject corrupted archives, parse a non-sensitive manifest, and use different salts/nonces across exports.
- The implementation does not import security modules from UI, collector, statistics, export, runtime startup, or database initialization paths.
- Existing v0.1 validation, CI, PyInstaller build, and installer build remain intact.
