# WorkTrace v0.2 Local Security Design

## Phase 1A Scope

This phase establishes the local security foundation for later v0.2 work. It adds:

- local security foundation;
- encrypted backup/import foundation;
- Windows current-user DPAPI keyring;
- AEAD envelope format;
- `.wtbackup` encrypted backup container foundation;
- no AI;
- no server;
- no payment, license, token, or subscription feature;
- no SQLCipher;
- no runtime DB field encryption yet.

Phase 1A does not change the existing runtime database schema, startup flow, collector behavior, UI behavior, statistics, export, or user-data write path.

## Route

The local security work is split into small phases:

1. Phase 1A: crypto foundation, DPAPI keyring, and `.wtbackup` format.
2. Phase 1B: encrypted backup/export/import integration.
3. Phase 1C: first batch high-sensitive field encryption.
4. Phase 1D: project, rule, folder-index encryption and blind indexes.

## SQLCipher Decision

SQLCipher is not adopted in Phase 1A.

The current Python, Windows, and PyInstaller distribution chain is uncertain for SQLCipher. Unofficial wheels are not suitable as a formal WorkTrace runtime dependency, and maintaining a custom SQLCipher wheel would add a high maintenance burden. A SQLCipher spike may remain useful for future investigation, but it is not part of this phase.

## Runtime Database Boundary

Phase 1A does not modify the existing runtime database. It does not add encrypted database columns, change existing business fields, require startup migration, change collection writes, or change how existing WorkTrace user data is stored during normal app operation.

The new security modules are independent foundations for future integration.

## Cryptography

The AEAD envelope uses AES-GCM with a fresh 96-bit random nonce for each encryption operation. Callers cannot provide a nonce. The envelope is versioned as `wtenc1:<base64url-json>` and contains only encryption metadata and ciphertext.

Backup passphrase derivation targets Argon2id as the preferred long-term design. Phase 1A uses scrypt as the packaging-conservative fallback because it is available through `cryptography`, keeps the dependency set small, and is more predictable for Windows/PyInstaller than adding Argon2 bindings at this stage. Backup manifests serialize the KDF algorithm, parameters, and unique random salt so the format can migrate later.

## Security Goals

- Backup files cannot be viewed as plaintext.
- The keyring never stores plaintext data keys.
- Encryption failure must not silently downgrade to plaintext.
- A wrong backup password must not decrypt.
- Nonces must not be reused by the encryption API.
- AAD mismatch must fail.
- Logs and errors must not include keys, passwords, plaintext sensitive content, or complete ciphertext.

## Not Protected

Phase 1A does not protect against:

- malware running as the same Windows user;
- runtime memory capture;
- plaintext Excel files that the user actively exports.

## Acceptance Criteria

Phase 1A is accepted when:

- `worktrace/security` contains independent crypto, KDF, key manager, and `.wtbackup` modules.
- AES-GCM round trips work and wrong keys, wrong AAD, corrupted ciphertext, and malformed envelopes fail.
- Re-encrypting the same plaintext produces different envelopes.
- The DPAPI keyring stores wrapped data keys at `%LOCALAPPDATA%\WorkTrace\security\keyring.json` and never stores plaintext data keys.
- Non-Windows and tests can use a fake key wrapper without real DPAPI.
- `.wtbackup` can encrypt/decrypt a payload with a passphrase, reject wrong passphrases, reject corrupted archives, parse a non-sensitive manifest, and use different salts/nonces across exports.
- The implementation does not import security modules from UI, collector, statistics, export, runtime startup, or database initialization paths.
- Existing v0.1 validation, CI, PyInstaller build, and installer build remain intact.

## Phase 1B Scope

Phase 1B connects the Phase 1A `.wtbackup` encrypted container to the real WorkTrace local database export and import paths. It adds:

- real local database encrypted backup export;
- real local database encrypted backup import;
- Settings/Privacy UI entry points for encrypted backup;
- API/service boundary so the UI never imports `worktrace.security.crypto`, `worktrace.security.kdf`, or `worktrace.security.backup_format` directly;
- tests covering payload export, encrypted export, encrypted import, API/UI boundary, and regression;
- release validation updates.

Phase 1B still does not:

- perform runtime field-level encryption;
- introduce SQLCipher;
- change collector behavior;
- change normal SQLite read/write behavior;
- add network dependencies;
- require administrator privileges;
- auto-upload `.wtbackup` anywhere;
- change Excel export plaintext user-active export semantics;
- perform merge import (only replace import or import into an empty profile);
- log passphrase, decrypted payload, window title, path, note, copied text, or full ciphertext.

### `.wtbackup` File Semantics

A `.wtbackup` file is a user-initiated encrypted migration/backup file. It:

- is created only when the user explicitly chooses to export an encrypted backup;
- does not contain the keyring;
- does not contain logs;
- does not contain `folder_rule_file_index` derived cache by default;
- may contain clipboard events, because the file is encrypted; the UI warns the user before export that the backup may include copied text;
- uses a passphrase chosen by the user at export time;
- cannot be recovered if the passphrase is forgotten.

### Payload Format

The encrypted payload is UTF-8 JSON with this top-level shape:

```json
{
  "format": "worktrace-local-data",
  "version": 1,
  "created_at": "<ISO 8601 UTC>",
  "app_version": "<WorkTrace version>",
  "schema_version": "<schema.sql version marker>",
  "tables": {
    "project": [...],
    "activity_log": [...],
    "settings": [...],
    "session_boundary": [...],
    "folder_project_rule": [...],
    "folder_rule_index_state": [...],
    "project_rule": [...],
    "activity_project_assignment": [...],
    "activity_clipboard_event": [...],
    "project_session_note": [...],
    "activity_resource": [...]
  }
}
```

Excluded by default:

- `logs`;
- `keyring`;
- `folder_rule_file_index` derived cache;
- any build/dist artifacts.

### Import Semantics

Phase 1B import supports only:

- replace import: the current local database is replaced by the backup content;
- import into an empty profile.

Phase 1B import does not support merge import.

Import requirements:

- the caller must ensure the app is in a safe state (collector paused or stopped) before invoking import;
- import runs inside a single transaction;
- import failure rolls back and leaves the current database intact;
- wrong passphrase, corrupted backup, or unsupported version never clear or partially overwrite the current database;
- after a successful import, default rows are re-seeded if missing;
- after a successful import, `folder_rule_file_index` is cleared and `folder_rule_index_state` is marked `pending` so the folder-index worker rebuilds it;
- import does not import the keyring; the local machine continues to use its existing keyring.

### Error Handling And Logging

- Wrong passphrase shows only a generic "could not decrypt backup or wrong passphrase" message.
- Corrupted backup shows "backup file is invalid or corrupted".
- Unsupported version shows "backup version is not supported".
- Logs record only operation type, result, and exception type. Logs never include passphrase, decrypted payload, window title, path, note, copied text, or full ciphertext.
- Import failure never clears or partially overwrites the current database.

### API Boundary

The UI talks to encrypted backup exclusively through `worktrace.api`. The UI must not import:

- `worktrace.security.crypto`;
- `worktrace.security.kdf`;
- `worktrace.security.backup_format`.

The API surface includes:

- `export_encrypted_backup(output_path, passphrase) -> Path`;
- `import_encrypted_backup(input_path, passphrase, mode="replace") -> ImportResult`;
- `parse_encrypted_backup_manifest(input_path) -> BackupManifest`.

### Phase 1B Acceptance Criteria

Phase 1B is accepted when:

- A real WorkTrace database can be exported to a `.wtbackup` file.
- The `.wtbackup` file does not contain plaintext project names, window titles, file paths, notes, or copied text.
- A correct passphrase imports the backup.
- A wrong passphrase fails without damaging the current database.
- A corrupted backup fails without damaging the current database.
- An unsupported version fails without damaging the current database.
- Replace import restores projects, activities, resources, notes, and clipboard events.
- `folder_rule_file_index` is not imported and is left in a rebuildable state.
- The UI exposes export/import through Settings/Privacy without importing security modules directly.
- `pytest`, PyInstaller build, and installer build remain intact.
- No network dependency and no administrator privilege is required.
