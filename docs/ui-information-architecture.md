# WorkTrace UI 信息架构与低保真设计

## 1. 文档状态

- 阶段：UI 重构阶段 2。
- 性质：目标信息架构和低保真设计基线。
- 前置基线：`docs/ui-current-state-audit.md`。
- 适用范围：Overview、Timeline、Statistics / Export、Project Rules、Settings / Privacy，以及全局应用框架。
- 本阶段不修改正式 HTML、CSS、JavaScript、Python、bridge、DTO、数据库、测试或打包配置。

本文决定“页面应如何组织和交互”，不决定最终颜色、阴影、图标细节和精确像素。阶段 3 的设计系统必须服从本文；后续确有产品事实变化时，应显式修订本文，不得在页面实现中悄然偏离。

---

## 2. 设计目标

WorkTrace 是面向律师和知识工作者的本地 Windows 时间记录工具。UI 应呈现以下特征：

- 专业、克制、可信；
- 中等信息密度，优先支持快速扫描；
- 接近成熟 Windows 生产力软件，而非营销型网页仪表盘；
- 以时间、项目和异常处理为中心；
- 清楚区分查看、普通编辑、结构调整和危险操作；
- 在实时更新时保持页面稳定，不产生可见跳动；
- 不要求用户理解 Collector、generation、revision、maintenance latch 等内部概念。

明确避免：

- 大面积渐变、玻璃拟态和高饱和配色；
- 每个内容区都使用独立卡片；
- 无业务意义的动画；
- 为展示效果引入新的统计数据、后端接口或图表框架；
- 将 Timeline 设计成复杂项目管理或拖拽排程工具；
- 将内部诊断状态直接作为普通设置展示。

---

## 3. 不可突破的产品与技术边界

阶段 2 的所有设计只重新组织已有页面模型和操作，不新增业务语义。

必须保持：

1. Overview 继续拥有当前活动的主要呈现。
2. Recent 第一条与当前活动使用既有运行时身份和时间语义。
3. Timeline 只消费后端已经物化的时段、详情和编辑能力。
4. UI 不自行推断归属、可见性、统计、开放活动物化或 LiveClock 候选。
5. Statistics / Export 使用后端接受的统计事实和 export revision。
6. Project Rules 与 Timeline 继续共享既有项目目录协调逻辑。
7. Settings 的备份、导入、清空和隐私确认契约不变。
8. 不改变 bridge 方法、DTO、DOM 实时计时属性或数据世代重置规则。
9. 所有资源保持本地，不增加网络、浏览器存储或前端构建链。
10. 不引入 React、Vue、Vite、Electron、Tauri、Redux 类状态容器或本地 HTTP 服务。

允许在后续实现阶段进行的表现层调整：

- 调整 DOM 层级；
- 保留稳定 ID 的前提下增加语义元素和测试锚点；
- 将点击式 `div` 改为可键盘操作的元素；
- 将低频操作移入菜单或 Drawer；
- 对现有页面数据进行纯展示排序、筛选和折叠；
- 将内部状态转换为面向用户的摘要，同时在 Advanced 中保留诊断详情。

## 4. 目标应用框架

### 4.1 全局结构

```text
┌──────────────┬────────────────────────────────────────────────────────────┐
│ App navigation│ Page header                                                 │
│              ├────────────────────────────────────────────────────────────┤
│              │ Page toolbar / contextual controls                           │
│              ├────────────────────────────────────────────────────────────┤
│              │                                                             │
│              │ Primary page workspace                                      │
│              │                                                             │
│              ├────────────────────────────────────────────────────────────┤
│ Collector    │ Inline status / transient feedback when needed              │
│ state + pause│                                                             │
└──────────────┴────────────────────────────────────────────────────────────┘
```

全局框架只包含：

- 品牌；
- 五个一级导航；
- Collector 面向用户的状态；
- pause / resume 主操作；
- 当前页面标题、说明和页面级主要操作。

不得在全局导航中加入项目、日期、导出或设置页内部入口。

### 4.2 导航模式

#### 标准模式：窗口宽度不小于 960px

- 左侧导航宽度建议 192–208px；
- 同时显示图标和文字；
- Collector 状态与 pause / resume 固定在底部；
- 当前项使用背景、文字和 `aria-current="page"` 共同表达；
- 不使用仅靠颜色区分的细线高亮。

#### 紧凑模式：窗口宽度小于 960px

- 左侧导航收缩为 56–64px 的 rail；
- 保留图标，文字通过 Tooltip 或可访问名称提供；
- 不使用汉堡菜单覆盖主要内容，因为五个一级页面数量固定且需要快速切换；
- Collector 状态用状态点和 pause / resume 图标按钮表达；
- 所有图标按钮必须有可访问名称。

### 4.3 页面标题栏

页面标题栏应统一包括：

- 页面名称；
- 一行可选说明；
- 当前页面最重要的一至两个操作。

页面标题栏不应放入大量筛选器。日期、搜索和筛选应进入紧邻内容的页面工具栏。

### 4.4 滚动责任

- 左侧导航不随页面内容滚动；
- 页面标题栏可保持普通文档流，不强制 sticky；
- Timeline 的工具栏和 Inspector 可在标准窗口中 sticky；
- Drawer 和 Dialog 自己管理滚动；
- 页面不得同时形成三个以上相互竞争的纵向滚动容器。

---

## 5. 尺寸与响应式约束

必须设计和验收：

| 场景 | 窗口尺寸 / 缩放 | 目标 |
|---|---:|---|
| 默认窗口 | 1080×720 | 完整标准工作区 |
| 最小窗口 | 800×540 | 所有关键操作可达，不发生横向页面滚动 |
| 常见笔记本 | 1366×768 | 信息密度自然，不显空旷 |
| 大屏 | 1920×1080 | 内容设置合理最大宽度，避免无限拉伸 |
| Windows 缩放 | 125%、150% | 文字、按钮、Drawer 和表格不被裁切 |

响应式规则：

1. 960px 以下使用紧凑导航 rail。
2. Timeline 常驻 Inspector 仅在内容宽度足够时显示；不足时使用右侧覆盖 Drawer。
3. Statistics 的次级统计不并排放置三张窄表，而使用 Tabs 或纵向区块。
4. Settings 在窄窗口中取消页面内双栏导航，改为单列分组。
5. 表格可以在自身容器内横向滚动，但整个页面不得横向滚动。
6. 任何操作不得只在 hover 时出现。

## 6. 全局信息层级

### 一级：当前状态与主要任务

- 当前是否正在记录；
- 当前活动；
- 今日时间；
- 当前选择的时段；
- 当前页面主要操作。

### 二级：分析和管理

- 最近活动；
- 项目时间；
- 规则列表；
- 统计分组；
- 设置分组。

### 三级：诊断、帮助和低频操作

- Collector 原始状态；
- maintenance / recovery 细节；
- 结构编辑菜单；
- 备份清单技术字段；
- 删除历史归属选项。

不得让三级信息与一级信息获得相同视觉权重。

---

## 7. 通用交互模式

### 7.1 页面级操作

- 每个页面最多一个主要按钮；
- 第二个常用操作使用次按钮；
- 低频操作进入 `更多` 菜单；
- 危险操作不与主要按钮并排使用相同视觉权重。

### 7.2 选中与详情

- 桌面宽度充足时使用列表 / 时间轴 + 右侧 Inspector；
- 窄窗口使用覆盖式 Drawer；
- 切换选中项时不得重建整个页面；
- 编辑未保存时，实时刷新不得覆盖输入；
- Inspector 关闭后焦点返回原选中项。

### 7.3 状态反馈

统一为：

- Inline Alert：页面级加载失败、阻断或需用户处理的问题；
- Field Error：字段校验；
- Inline Status：保存中、导出中等与当前操作绑定的状态；
- Toast：操作已经完成且无需保留在页面中的通知；
- Empty State：无数据；
- Blocking Dialog：隐私首次确认和不可绕过错误。

动态状态使用 `role="status"` 或适当的 `aria-live`。错误使用 `role="alert"`，但不得让每秒实时计时触发播报。

### 7.4 Dialog 风险分级

- 普通删除：Confirm Dialog；
- 删除规则并可选处理历史归属：带说明和选项的 Dialog；
- 导入并替换数据库、清空数据：继续要求输入确认文字；
- 不再使用 `window.confirm`。

# 8. Overview 信息架构

## 8.1 页面职责

Overview 只回答四个问题：

1. WorkTrace 现在是否正常记录？
2. 当前正在做什么、已经持续多久？
3. 今天总共记录了多少时间，其中多少已归类？
4. 最近做了什么，是否有需要处理的未归类时间？

Overview 不承担完整统计、规则管理或批量纠错。

- 项目分布只能使用当前页面模型已有数据；如 Overview DTO 尚无项目汇总，不在 UI 阶段临时计算或新增 API，应先保留 Recent 与未归类入口，项目分布延后；
- 未归类入口只导航到既有 Timeline 能支持的范围，不在阶段 2 定义新的过滤业务语义。

# 9. Timeline 信息架构

## 9.1 页面职责

Timeline 是 WorkTrace 的主要工作台，负责：

- 查看某一天的时段；
- 识别当前进行中活动；
- 选择一个时段查看活动耗时；
- 在既有能力范围内修改项目、时长和备注；
- 执行已有的复制、合并、拆分、隐藏等结构操作。

它不负责：

- 拖拽排程；
- 任意编辑真实开始和结束时间；
- 前端重新归并活动；
- 推断项目或生成候选归属。


窄窗口的详情为右侧覆盖 Drawer，不把详情追加到列表底部。

## 9.4 工具栏

标准宽度：

- 日期导航；
- 今日；
- 项目筛选；
- 状态筛选；
- 当日总时长。

窄宽度：

- 保留日期导航、今日和总时长；
- 项目与状态筛选收纳到一个 `筛选` 弹出层；
- 筛选仅作用于后端已经返回的当前页面行；若筛选会改变统计语义或需要新 API，则延后。

## 9.5 时段行

每行只显示：

- 开始时间；
- 项目；
- 进行中 / 已修正等必要状态；
- 时长；
- 活动条数作为低权重信息。

不得默认显示完整窗口标题、路径或其他敏感字段。

进行中状态使用：

- 小状态点或轻量边框；
- “进行中”文字；
- 实时时长。

不得依赖高饱和整行底色作为唯一标识。

## 9.6 Inspector

Inspector 按以下顺序组织：

1. 时段时间、项目和总时长；
2. 活动耗时摘要；
3. 可编辑字段；
4. 保存 / 取消；
5. 更多结构操作。

“删除、合并到上一时段、合并到下一时段、拆分、复制”进入 `更多` 菜单。对于当前能力不允许的操作，使用禁用状态和简短原因，不隐藏后造成用户误解。

## 9.7 键盘行为

- 上下方向键在时段间移动；
- Enter 或 Space 选择；
- Escape 关闭窄窗口 Drawer；
- Drawer 关闭后返回原时段；
- `Ctrl+S` 可作为编辑保存的增强项，但不是阶段 3 必须项；
- 实时计时不得触发屏幕阅读器每秒播报。

## 9.8 实时更新限制

- 每秒仅更新已有时长节点；
- 不重建时段列表、Inspector 或项目选择框；
- 页面结构 revision 变化时才重新读取并接受页面模型；
- 编辑为 dirty 或 saving 时，刷新不得覆盖表单；
- 设计不得要求第二套本地时钟或页面缓存。

# 10. Statistics / Export 信息架构

## 10.1 页面职责

回答：

- 选择范围内总共记录多少时间；
- 时间主要分布在哪些项目；
- 需要时如何导出当前已接受的统计快照。

## 10.2 线框

```text
┌───────────────────────────────────────────────────────────────────────────┐
│ 统计与导出                                             [导出 CSV]          │
│ 统计仅包含已完成活动                                                       │
├───────────────────────────────────────────────────────────────────────────┤
│ [开始日期] 至 [结束日期] [加载]     [今天] [最近 7 天] [本月]              │
├───────────────────────────────────────────────────────────────────────────┤
│ 总时长 42:16:08   活动 286   项目 12   应用 18                            │
├───────────────────────────────────────────────────────────────────────────┤
│ [按项目] [按应用] [按状态]                                                 │
│ 名称                              时长          活动数          占比        │
│ 项目 A                            12:30:10      48              29.6%       │
│ 项目 B                            09:18:22      35              22.0%       │
│ ...                                                                       │
├───────────────────────────────────────────────────────────────────────────┤
│ 导出范围与隐私说明                                      [展开导出详情 ▾]   │
└───────────────────────────────────────────────────────────────────────────┘
```

## 10.3 设计决策

- 不再并排展示三张窄表；使用 Tabs 共享一个主表；
- “按项目”为默认页签；
- 不新增趋势图，因为当前 DTO 未确认提供按日序列；
- 导出预览折叠为次级区域，导出按钮仍使用已接受的 export revision；
- “仅包含已完成活动”的提示只出现一次；
- 日期改变后导出按钮保持失效，直至新统计被接受；
- 表格排序若实现，只对当前已返回数据进行纯展示排序，不改变导出顺序或统计事实。

# 11. Project Rules 信息架构

## 11.1 页面职责

- 查看项目及其规则；
- 创建和编辑普通项目；
- 创建文件夹或关键词规则；
- 删除项目或规则；
- 选择创建规则后是否应用到历史记录。

## 11.2 线框

```text
┌───────────────────────────────────────────────────────────────────────────┐
│ 项目规则                              [新建规则] [新建项目]                │
│ 自动将活动归入项目                                                       │
```

# 13. 页面状态线框

所有页面必须显式设计以下状态。

## Loading

- 保留页面框架和主要控件位置；
- 首次加载可使用少量 Skeleton；
- 刷新时优先保留旧数据并显示局部“正在刷新”；
- 不整页闪白。

## Empty

- 一句说明；
- 最多一个合理动作；
- 不将空状态包装成大型营销插图。

例：

```text
当日暂无活动记录
WorkTrace 开始记录后，活动会显示在这里。
```

## Error

- 说明用户可理解的失败；
- 可重试时提供重试；
- 不展示 traceback、异常类型或敏感路径；
- 有旧数据时不清空旧数据。

## Saving / Unknown

- pending：禁用冲突操作并显示“正在保存”；
- unknown：保留当前选择和输入，提示“结果尚未确认，可重试同一操作或刷新核对”；
- 不把 unknown 视觉上伪装为普通失败。

## Blocked

- 隐私首次确认失败或恢复阻断属于 Blocking 状态；
- 明确说明记录没有启动或仍被阻断；
- 提供当前确实可执行的恢复路径，不提供无效的继续按钮。

# 14. 可访问性基线

阶段 3 和页面实施必须满足：

- 所有交互均可通过键盘到达；
- 统一 `:focus-visible`；
- 当前导航设置 `aria-current="page"`；
- Timeline 选中项具备列表项和选中语义；
- Tabs 使用正确的 tab / tabpanel 关系；
- Drawer、Dialog 使用正确名称、模态状态和焦点管理；
- 动态状态使用适当的 `aria-live`；
- 图标按钮具有可访问名称和 Tooltip；
- 不能仅通过颜色表达进行中、错误、危险或选中；
- 正文和控件文字在 150% 缩放下不被裁切；
- 表格标题和行列关系清晰。

# 16. 验收标准

阶段 2 通过需满足：

1. 五个页面职责不重叠且没有遗漏现有主要能力。
2. 线框不依赖未存在的后端字段或 API；任何条件性建议均明确标注延后。
3. 1080×720 和 800×540 均有清晰布局策略。
4. Timeline 在标准窗口使用 Inspector，在窄窗口使用 Drawer。
5. 低频结构操作不再挤占主要编辑区。
6. Statistics 不再并排显示三张窄表。
7. Rules 不再依赖项目卡片中重复的大量按钮。
8. Settings 将内部诊断与普通设置分层。
9. 可访问性、Loading、Empty、Error、Unknown 和 Blocking 状态均有设计约束。
10. 未改变 LiveClock、bridge、DTO、统计、导出、规则、隐私或数据库语义。

# 17. 阶段结论

阶段 2 的目标信息架构已经确定：

- 保留五个一级页面；
- 保留左侧桌面导航，在窄窗口收缩为 rail；
- Overview 强化当前活动与今日时间；
- Timeline 成为带时间刻度列表和 Inspector 的主要工作台；
- Statistics 使用单一主表和 Tabs；
- Project Rules 使用紧凑可展开项目列表与统一 Drawer；
- Settings 采用页面内分类，并将技术状态收纳到 Advanced；
- 所有页面使用统一的状态、危险操作和键盘交互模式。

本文完成后，可以进入阶段 3：建立最小设计系统与全局应用框架。正式实施前，应先使用静态低保真原型检查默认窗口和最小窗口的空间分配。
