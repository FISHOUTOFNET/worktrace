# WorkTrace UI 信息架构与低保真设计

## 1. 文档状态

- 阶段：UI 重构阶段 2。
- 性质：目标信息架构和低保真设计定稿。
- 前置基线：`docs/ui-current-state-audit.md`。
- 适用范围：Overview、Timeline、Statistics / Export、Project Rules、Settings / Privacy，以及全局应用框架。
- 本文只确定页面职责、信息层级、主要交互、响应式布局、状态反馈和技术边界。
- 本文不确定最终配色、阴影、图标、动画和精确像素。

阶段 3 的设计系统和后续页面实现必须服从本文。后续如产品事实变化，应先显式修订本文，不得在实现中悄然偏离。

---

## 2. 产品定位与设计原则

WorkTrace 是本地 Windows 时间记录工具。

首个主要应用场景是律师及其他知识工作者的时间记录，但产品设计不应将其限定为工作用途。

### 2.1 目标体验

UI 应当：

- 专业、克制、可信；
- 保持中等信息密度；
- 支持快速扫描和低成本整理；
- 接近成熟 Windows 生产力软件，而非营销型网页仪表盘；
- 以时间、项目、描述和活动明细为核心；
- 清楚区分用户输入、系统事实和系统派生信息；
- 在实时更新和自动保存时保持页面稳定；
- 对高频操作保持直接；
- 对低频操作降低视觉权重；
- 对永久删除提供与风险相匹配的确认；
- 不要求用户理解内部技术状态。

### 2.2 明确避免

不得采用：

- 大面积渐变、玻璃拟态和高饱和装饰色；
- 每个内容区都使用独立卡片；
- 无业务意义的动画；
- 为展示效果增加无必要的统计、接口或图表；
- 传统拖拽排程式 Timeline；
- 在多个页面重复相同统计；
- 自动刷新页面中的普通刷新按钮；
- 常用字段的显式保存和取消按钮；
- 将内部状态直接暴露给普通用户；
- 将系统生成的摘要冒充用户填写的描述；
- 将常用删除入口隐藏到多层菜单；
- 为极少使用的能力长期占据界面空间。

## 3. 核心术语和数据语义

### 3.1 时间段

“时间段”是 Timeline 左侧列表中的主要记录单位。

每个时间段可以包含：

- 项目；
- 时长；
- 用户描述；
- 多条活动明细。

Timeline 左侧展示的是时间段，不称为“活动列表”。

### 3.2 活动明细

“活动明细”是构成某个时间段的底层活动信息，例如：

- 应用名称；
- 窗口标题；
- 文件名称；
- 页面标题；
- 活动持续时间。

活动明细显示在 Timeline 右侧编辑区域下方。

时间段和活动明细是不同层级的对象：

- 删除时间段会影响整个时间段；
- 删除活动只影响对应活动明细；
- 两种删除必须使用不同文案和稳定目标身份。

### 3.3 时间段描述

“描述”是用户对某个时间段内容的主动概述。

时间段描述：

- 由用户填写；
- 可以为空；
- 可以编辑；
- 自动保存；
- 属于用户数据；
- 可以进入现有描述字段和相应导出；
- 不限定为工作用途。

界面字段名称统一使用“描述”，不使用“备注”“工作描述”或“申报说明”。

### 3.4 项目描述

Project Rules 中的项目也可以具有描述。

项目描述：

- 由用户填写；
- 可以为空；
- 用于解释项目内容或背景；
- 与时间段描述是不同实体上的字段；
- 不根据规则或活动自动生成；
- 不使用派生摘要。

设计文档中分别称为“时间段描述”和“项目描述”，实际界面字段均显示为“描述”。

### 3.5 派生摘要

当用户没有填写时间段描述时，系统为展示目的生成派生摘要。

派生摘要生成规则：

1. 仅使用当前时间段内的活动明细；
2. 按活动持续时间从长到短排列；
3. 取最长的三个不同活动标题；
4. 忽略空标题；
5. 合并重复标题；
6. 少于三个标题时，只显示已有标题；
7. 没有可显示标题时，显示“暂无描述”；
8. 继续遵守现有隐私、排除和敏感字段展示规则。

派生摘要：

- 不写入时间段描述字段；
- 不触发自动保存；
- 不进入编辑历史；
- 不作为用户描述导出；
- 不参与描述字段是否已填写的判断；
- 用户填写描述后立即停止显示；
- 用户清空描述并保存成功后重新显示。

### 3.6 描述统一显示规则

Overview 当前活动、Overview 待整理、Overview 最近记录和 Timeline 时间段列表统一使用：

```text
存在用户描述
→ 显示用户描述

用户描述为空，存在活动标题
→ 显示派生摘要

用户描述为空，不存在可展示活动标题
→ 显示“暂无描述”
```

派生摘要使用较弱视觉样式，并显示轻量的“自动摘要”标识。

## 4. 全局应用框架

### 4.1 基本结构

```text
┌──────────────┬────────────────────────────────────────────────────────────┐
│ 主导航       │ 页面标题                                                   │
│              ├────────────────────────────────────────────────────────────┤
│              │ 页面工具栏                                                 │
│              ├────────────────────────────────────────────────────────────┤
│              │                                                            │
│              │ 页面工作区                                                 │
│              │                                                            │
│              ├────────────────────────────────────────────────────────────┤
│ 记录状态     │ 局部状态或操作反馈                                         │
│ 暂停 / 恢复  │                                                            │
└──────────────┴────────────────────────────────────────────────────────────┘
```

### 4.2 一级导航

保留五个一级页面：

- Overview；
- Timeline；
- Statistics / Export；
- Project Rules；
- Settings / Privacy。

不得在主导航中加入项目、日期、导出、设置子页面或普通刷新入口。

### 4.3 标准导航

窗口宽度不小于 960px 时：

- 左侧导航宽度建议 192–208px；
- 同时显示图标和文字；
- 记录状态与暂停 / 恢复位于底部；
- 当前项使用背景、文字和 `aria-current="page"` 表达；
- 不仅依赖颜色表达当前页面。

### 4.4 紧凑导航

窗口宽度小于 960px 时：

- 左侧导航收缩为 56–64px rail；
- 保留图标；
- 使用 Tooltip 和可访问名称提供文字；
- 不使用汉堡菜单隐藏五个一级页面；
- 记录状态使用状态点；
- 暂停 / 恢复使用图标按钮。

### 4.5 页面标题

页面标题区域只包括：

- 页面名称；
- 一行可选说明；
- 页面确有必要的主要操作。

不得机械加入刷新、重新加载或与页面主要任务无关的次级操作。

### 4.6 自动刷新原则

Overview、Timeline、Project Rules 和 Settings 依赖既有自动刷新与状态协调。

删除：

- Overview 刷新按钮；
- Timeline 刷新按钮；
- Project Rules 刷新按钮；
- Settings 状态刷新按钮。

保留：

- 错误后的重试；
- 用户改变 Statistics 查询条件后自动发起的新查询；
- 导入、恢复或维护失败后的必要恢复入口。

### 4.7 正常状态去重

普通用户不需要看到：

- idle；
- error；
- excluded；
- suppressed；
- generation；
- revision；
- maintenance latch；
- recovery token。

全局导航底部已经显示记录状态时，Overview 不再重复显示“记录正常”。

仅在存在需要用户处理的问题时显示：

- 已暂停；
- 记录服务异常；
- 隐私确认未完成；
- 恢复尚未完成；
- 记录被阻断。

## 5. 尺寸与响应式约束

必须验证：

| 场景 | 尺寸或缩放 | 目标 |
|---|---:|---|
| 默认窗口 | 1080×720 | 完整标准工作区 |
| 最小窗口 | 800×540 | 所有核心操作可达 |
| 常见笔记本 | 1366×768 | 信息密度自然 |
| 大屏 | 1920×1080 | 内容不无限拉伸 |
| Windows 缩放 | 125%、150% | 文字和控件不被裁切 |

通用规则：

1. 960px 以下使用紧凑导航 rail。
2. 整个页面不得出现横向滚动。
3. 表格可以在自身容器内横向滚动。
4. 任何操作不得仅在 hover 时出现。
5. 描述和派生摘要可以截断，但不能完全隐藏。
6. Dialog 和 Drawer 自己管理滚动。
7. 页面不得形成三个以上互相竞争的纵向滚动区域。

# 6. Overview 信息架构

## 6.1 页面职责

Overview 围绕三个核心模块组织：

1. 当前活动：用户此刻具体在做什么（原子级活动快照）？
2. 最近记录：最近的时间被如何归类和合并（报告时间段）？
3. 待整理：哪些已经结束的记录需要用户采取操作？

此外保留一个次级“今日记录摘要”条，用于呈现今日总时长、已归类时长、
未归类时长和涉及项目数。该摘要条是次级对象，不与三个核心模块竞争视觉
权重，也不替代它们。

Overview 不承担：

- 项目维度汇总；
- 完整统计；
- 规则管理；
- 批量纠错；
- 导出；
- 手动刷新；
- 项目、时长或描述的直接编辑。

## 6.2 页面内容

Overview 保留：

- 今日记录摘要条（今日总时长、已归类、未归类、涉及项目数）；
- 当前活动；
- 最近记录；
- 待整理。

摘要条的统计口径来自完整权威投影，不得从最近记录的显示截断反向计算。

## 6.3 模块集合关系

三个核心模块的集合关系固定为：

```text
待整理 ⊆ 最近记录
```

- 最近记录包含所有符合现有报告可见性策略的报告时间段，包括已整理记录、
  待整理记录和符合策略的当前进行中时间段；
- 待整理是最近记录中已经结束且缺少项目或用户描述的可操作子集，不是与
  最近记录互斥的分区；
- 当前活动展示的是当前原子级活动快照，不属于“最近记录”的合并报告粒度；
  当前进行中的报告时间段可以同时既出现在“当前活动”背后的 current_session
  中，也作为最近记录的第一条显示。

该子集关系同时约束实际 payload：Overview 返回给前端的可见 attention 项
必须全部包含在当前可见 recent 列表中，即使两者分别被截断到各自的展示上限
（recent 上限 20，attention 上限 3）。ViewModel 选择展示窗口时优先保留
进行中记录，再以尾部普通已整理记录替换为超出截断边界的待整理项，从而
保证 visible attention ⊆ visible recent。

不得重新引入“三组互斥”或“去重展示”的过滤分支。底层页面模型保持
current_session 与 recent 第一条使用既有身份语义，UI 不通过互斥过滤改变
它们的可见性。

## 6.4 待整理定义

“待整理”只包含今天已经结束、当前可在 Timeline 中显示，并满足下列至少一项的时间段：

- 未选择项目；
- 用户描述为空。

派生摘要不能使时间段被视为已填写描述。

以下内容不进入待整理：

- 当前仍在进行中的时间段；
- 后端已抑制或不允许前端显示的内容；
- 已有项目且已有用户描述的时间段。

待整理是现有显式字段的展示分类，不引入新的持久化状态。

## 6.5 待整理交互

每项显示：

- 开始时间；
- 当前项目或“未选择项目”；
- 时长；
- 缺失内容；
- 用户描述、派生摘要或“暂无描述”；
- “整理”入口。

点击“整理”：

- 打开 Timeline 对应日期和时间段；
- 没有项目时优先聚焦项目字段；
- 已有项目但没有描述时聚焦描述字段；
- 两者都缺少时先聚焦项目字段。

Overview 不直接编辑项目、时长或描述，避免复制 Timeline 的草稿、自动保存和错误处理逻辑。

待整理最多显示三个时间段，按照时间从新到旧排列。

超过三个时显示：

```text
还有 4 条待整理记录                         [查看]
```

“查看”打开 Timeline 并选择最新的待整理时间段，不要求增加新的后端筛选语义。

没有待整理项时：

- 不保留大型空卡片；
- 显示一行轻量状态“今日记录已整理”；
- 最近记录扩展使用可用空间。

```text

│ 今日概览                                             今日 06:18:20         │
├───────────────────────────────────────────────────────────────────────────┤

├──────────────────────────────────────────┬────────────────────────────────┤
│ 最近记录                                 │ 待整理                         │
│                                          │                                │
│ 08:43  项目 B                00:36:40    │ 09:20  未选择项目   00:18:03  │
│        准备客户会议材料                  │        缺少项目和描述           │
│                                          │        自动摘要：专利检索 · 邮件│

│        审阅合同修改稿                    │                                │

│ 07:22  项目 E                00:20:06    │        缺少描述                 │
│        电话会议                          │        自动摘要：Excel · 浏览器 │
│                                          │                        [整理]  │
└──────────────────────────────────────────┴────────────────────────────────┘
```

当前活动的主标题必须是当前资源或活动名称（如 `overview.js`），不能是项目名称。
项目只作为次级归属信息出现在同一行（例如 `Visual Studio Code · 项目：WorkTrace`）。
未归类时显示 `Visual Studio Code · 未归类`。当前活动持续时间使用 `current_live`。

最近记录中当前进行中的报告时间段位于第一条，并显示“进行中”状态；待整理记录
也保留在最近记录中，并显示“待整理”状态。最近记录时长使用 `aggregate_live` 或
关闭后的 durable duration，允许与当前活动时长不同。

标准窗口下：

- 最近记录约占 60%–65%；
- 待整理约占 35%–40%；
- 当前活动始终独占上方完整宽度。

1. 当前活动；

## 6.8 当前活动

当前活动展示当前原子级活动快照，回答“用户此刻具体在做什么”。

主标题必须是当前资源或活动名称，来源于结构化 DTO 的 `resource_name` 字段，
不能是项目名称，也不能是归类后时间段的用户描述或自动摘要。

次级信息由应用名称和项目归属组成：

- 应用名称；
- 项目名称或“未归类”；
- 当资源名与应用名相同时，不重复显示相同文字。

当前活动持续时间只表示当前原子活动持续时间，使用 `current_live` 语义，
不与最近记录中的 `aggregate_live` 强行对齐。

自动摘要、用户时间段描述不得出现在当前活动卡片。

非正常状态必须显式覆盖：无当前快照、已暂停、空闲、已排除和采集异常。
每种状态显示对应的状态标题和提示文案，不显示过期资源、项目或时长增长，
也不泄露已排除内容的真实窗口标题、路径、文件、网页或项目。

当前活动本身可以点击进入 Timeline 对应的 current_session 时间段，但仅在
存在 current_session、稳定 projection 身份且 Timeline 可以定位该时间段时
才可点击；否则必须真实禁用，不保留可点击视觉但没有处理函数。

## 6.9 最近记录

最近记录回答“最近的时间被如何归类和合并”，展示经过状态与隐私策略、
项目归属、时间段合并和报告投影后形成的最终报告时间段。不得展示一条条
底层窗口切换或原始采集活动。

最近记录包含所有符合现有报告可见性策略的报告时间段，包括：

- 已整理记录；
- 待整理记录；
- 符合报告展示策略的当前进行中时间段。

不得因为记录同时属于“待整理”或“当前进行中”而将其从最近记录中排除。

最近记录按业务顺序稳定排序：

1. 当前进行中的报告时间段优先；
2. 其余按开始时间倒序。

每项使用两层结构。

第一行：

- 开始时间；
- 项目名称或“未归类”；
- 合并后的记录时长；
- 必要状态（“进行中”或“待整理”）。

第二行：

- 用户描述；
- 自动摘要（必须保留“自动摘要”标识）；
- 或“暂无描述”。

要求：

- 描述最多显示一行；
- 派生摘要最多显示一行；
- 过长时截断；
- 不设置展开按钮；
- 状态不只通过颜色表达，必须有文字或等价可访问文本；
- 点击后打开对应 Timeline 时间段。

# 7. Timeline 信息架构

## 7.1 页面职责

Timeline 是主要时间整理工作台，负责：

- 查看某一天的时间段；
- 查看当前进行中的时间段；
- 按从新到旧的顺序定位时间段；
- 查看项目、时长和描述；
- 修改项目、时长和描述；
- 自动保存修改；
- 查看活动明细；
- 删除时间段；
- 删除活动明细；
- 执行拆分、合并、复制和隐藏等高级操作。

Timeline 不负责：

- 项目汇总统计；
- 按内部状态筛选；
- 展示 idle、error、excluded 或 suppressed；
- 拖拽排程；
- 前端重新归并活动；
- 推断归属；
- 显式保存和取消；
- 普通刷新。

```text
[‹] [07-22] [›] [今日] [项目 ▾]   06:18:20
```

保留日期导航、今日、项目筛选和当日总时长。

删除状态筛选和刷新按钮。

## 7.4 时间段列表

左侧是按照开始时间倒序排列的时间段列表，不使用传统自上而下的时间刻度。

顺序：

1. 当前进行中的时间段；
2. 最近结束的时间段；
3. 更早的时间段；
4. 当天最早的时间段。

每个时间段使用两层结构。

第一行显示开始时间、项目、时长；当前时间段可以显示“进行中”。

第二行显示用户描述、派生摘要或“暂无描述”。

不得显示 idle、error、excluded、suppressed、已修正、完整路径、完整活动详情或活动数量。

“进行中”是用户需要理解的运行事实，不属于可筛选状态。

## 7.5 右侧编辑区域

右侧不重复显示左侧已经清楚表达的开始时间、项目摘要、总时长摘要、“时间记录”标题或“时段详情”标题。

右侧按以下顺序组织：

1. 项目和时长；
2. 描述；
3. 自动保存状态、高级操作和删除时间段；
4. 活动详情。

### 项目和时长

```text
项目 [项目 A                    ▾]    时长 [42 分钟]
```

项目占据主要宽度，时长使用固定或受限宽度；修改后自动保存。

### 描述

```text
描述
[起草许可协议并核对目标专利……]
[                                ]
```

描述独占一行，使用多行输入，默认保持紧凑。

当描述为空时：

- 输入框实际值保持为空；
- 派生摘要可以作为灰色 placeholder；
- placeholder 不触发保存；
- placeholder 不算用户输入。

### 操作行

```text
已保存                      [高级操作] [删除时间段]
```

自动保存状态低显著显示；高级操作使用次级按钮；删除时间段直接可见，并使用明确文案。

## 7.6 自动保存

Timeline 不显示保存或取消按钮。

### 项目

选择变化后立即保存，使用既有项目目录和写入契约。

### 时长

输入完成、失焦或短暂停顿后保存；无效值不提交，并在字段附近显示校验错误。

### 描述

使用防抖自动保存，建议停止输入 500–800ms 后提交；不在每次键入时刷新页面。

### 保存协调

必须满足：

- 每个选中时间段只有一个草稿；
- 保存请求绑定时间段稳定身份；
- 较早响应不得覆盖较新输入；
- 保存失败保留输入；
- unknown outcome 不回退为旧值；
- generation 变化后旧草稿失效；
- 实时刷新不得覆盖 dirty、saving、failed 或 unknown 状态。

状态使用：

```text
保存中…
已保存
保存失败，重试
保存结果尚未确认
```

“已保存”只短暂显示。

## 7.7 高级操作

高级操作只包括：

- 拆分时间段；
- 合并到上一时间段；
- 合并到下一时间段；
- 复制；
- 隐藏。

不包括删除时间段、删除活动、项目修改、时长修改或描述修改。

高级操作面板使用当前选中时间段的稳定身份，关闭后焦点返回高级操作按钮。

## 7.8 删除时间段

“删除时间段”直接位于编辑区域操作行。

删除必须经过两步确认，但使用同一个 Dialog 的两个步骤。

第一步：

```text
删除这个时间段？

项目 A
10:14，42 分钟

该时间段及其中的活动将被永久删除。
此操作无法撤销。

[取消] [继续]
```

第二步：

```text
再次确认删除

即将永久删除：
项目 A · 10:14 · 42 分钟

[返回] [确认删除时间段]
```

要求：

- 第一步不修改数据；
- 第二步确认后才执行删除；
- 删除按钮绑定时间段稳定身份；
- 删除 pending 时不允许同目标并发保存或结构操作；
- 删除 unknown 时不提前移除时间段；
- 删除成功后接受后端页面模型；
- 不由前端自行扣减时间。

删除后优先选择视觉上相邻的下一项；删除最后一项时选择上一项；没有剩余时间段时显示右侧空状态。

## 7.9 活动详情

活动详情位于编辑区域下方，并始终显示。

不提供展开、收起、折叠或默认隐藏。

活动详情可以显示：

- 应用；
- 活动标题；
- 文件或页面摘要；
- 持续时间；
- 其他现有可展示字段；
- 删除活动入口。

活动详情不重复显示派生摘要。

## 7.10 删除活动

每条可删除活动直接显示“删除活动”或具有明确可访问名称的删除图标。

要求：

- 不仅在 hover 时出现；
- 绑定活动稳定身份；
- 不依赖列表位置；
- 后端不可删除时禁用或隐藏，并说明原因；
- 不误删除整个时间段。

删除活动同样使用单个 Dialog 内的两步确认。

删除成功后：

- 接受后端返回的新时间段模型；
- 更新时间段时长；
- 更新活动详情；
- 用户描述为空时重新生成派生摘要；
- 用户已有描述时不覆盖描述；
- 不由前端自行重算统计。

删除最后一条活动后如何处理空时间段，由后端契约决定。

## 7.11 窄窗口

选择时间段后打开右侧覆盖 Drawer。

```text
┌──────────────────────────────────────────────┐
│                                         [×] │
│ 项目 [项目 A              ▾]  时长 [42 分钟]│
│                                              │
│ 描述                                         │
│ [起草许可协议并核对目标专利……             ] │
│                                              │

├──────────────────────────────────────────────┤
│ 活动详情                                     │
│ Word 文档编辑                     00:31:10   │
│ 许可协议草稿.docx               [删除活动]  │
│ Edge 资料查询                     00:11:06   │
│ 专利检索页面                    [删除活动]  │
└──────────────────────────────────────────────┘
```

Drawer 不显示重复标题；编辑在上、详情在下；高级操作和删除时间段始终可达。

## 7.12 实时更新

必须保持：

- 每秒只更新已有时长节点；
- 不重建时间段列表；
- 不重建右侧编辑区；
- 不重建活动详情；
- 不重建项目选择框；
- 当前时间段持续记录时保持列表首位；
- 页面 revision 变化时才接受新模型；
- 自动保存和删除遵守 mutation owner；
- 旧响应不得覆盖新 generation；
- 派生摘要更新不得写入描述字段。

# 8. Statistics / Export 信息架构

## 8.1 页面职责

Statistics / Export 负责：

- 统计指定时间范围或全部时间；
- 统计全部项目或指定项目；
- 组合时间和项目条件；
- 按项目查看结果；
- 按应用查看结果；
- 导出当前已接受的统计快照。

不负责按内部状态统计，也不展示单条描述或派生摘要。

## 8.2 筛选维度

存在两个独立维度。

### 时间范围

可选：

- 全部时间；
- 自定义范围；
- 今天；
- 本周；
- 本月。

### 项目

可选：

- 全部项目；
- 某个正式项目；
- 未归类。

两个条件可以单独使用，也可以组合。

## 8.3 快捷范围

快捷按钮固定为今天、本周、本月，删除“最近 7 天”。

“本周”使用用户本地时间，周一为第一天，当前周结束日期为今天。

“本月”为本月第一天至今天。

快捷时间变化不清除项目筛选，项目变化不清除时间范围。

## 8.4 自动查询

Statistics 不设置“应用筛选”或“加载”按钮。

- 点击今天、本周或本月后立即查询；
- 选择项目后立即查询；
- 自定义开始和结束日期均有效时，在日期选择完成或短暂防抖后查询。

查询期间：

- 保留旧结果；
- 显示“更新中…”；
- 不整页清空；
- 导出按钮禁用；
- 新查询接受后替换旧结果。

自动查询必须使用既有 request owner，防止旧请求覆盖新筛选。

## 8.6 分析视角

只保留按项目和按应用。

删除按状态、状态筛选、状态数和状态表格。

筛选某个项目时，“按项目”可以继续显示该项目一行，以保持页面结构稳定。

## 8.7 摘要指标

未筛选具体项目时可以显示：

- 总时长；
- 时间段数；
- 项目数；
- 应用数。

筛选具体项目时显示：

- 项目名称；
- 总时长；
- 时间段数；
- 应用数。

不显示“项目数 1”。

## 8.8 项目筛选数据

项目筛选必须复用既有项目目录协调逻辑。

不得建立 Statistics 独立项目缓存、从 Timeline DOM 中提取项目、将未归类伪装为普通项目，或在前端对已聚合结果二次过滤。

## 8.9 导出

导出必须绑定当前已接受的查询快照，包括时间范围、项目条件、export revision 和其他现有导出条件。

查询更新中时导出按钮禁用，不允许页面显示新条件却导出旧条件。

描述列只导出用户描述，派生摘要不进入描述列。

# 9. Project Rules 信息架构

- 查看项目；

- 为具体项目新建规则；

- 项目描述；
- 上次使用时间；
- 累计时间；
- 新建规则；
- 编辑；

  客户 A 专利事务

## 9.5 项目操作

每个项目只有新建规则、编辑和删除三个项目级操作，全部直接显示，不放入“更多”。

窄窗口可以使用紧凑文字按钮、具有清晰可访问名称的图标按钮，或换行到第二行，但不能隐藏到菜单。

## 9.6 新建和编辑项目 Drawer

新建项目和编辑项目共用同一 Drawer 和同一表单组件。

字段只有：

- 项目名称；
- 描述。

两种模式只区分标题、初始值、主按钮文案和创建或更新 mutation，共用字段布局、校验、错误展示、焦点管理、Drawer 尺寸和关闭行为。

## 9.7 新建规则 Drawer

新建规则从具体项目行进入，Drawer 标题直接显示项目上下文，不显示目标项目选择框。

规则类型：

- 文件夹规则；
- 关键词规则。

文件夹规则默认包含全部子文件夹，不显示“包含子文件夹”选项。

如果底层仍支持 `include_subfolders=false`，新 UI 新建规则固定提交 `true`，不必删除底层兼容能力。

## 9.8 项目摘要数据

上次使用时间：

- 来自后端项目使用事实；
- 不由前端扫描 Timeline；
- 没有使用记录时显示“尚未使用”；
- 近期使用友好相对格式；
- 较早时间显示具体日期。

累计时间：

- 来自权威项目聚合；
- 不由前端自行计算；
- 使用与 Statistics 一致的项目时间语义；
- 没有记录时显示 `00:00:00`；
- 不受当前 Statistics 筛选影响。

## 9.9 搜索和排序

搜索范围：

- 项目名称；
- 项目描述；
- 规则展示文本。

排序至少支持：

- 最近使用；
- 累计时间；
- 项目名称。

默认使用最近使用。

## 9.10 删除项目

删除按钮直接显示。

确认方式根据实际影响决定：

- 只删除项目定义、不删除时间记录：使用一次说明充分的确认；
- 永久删除相关时间数据：使用不低于删除时间段的两步确认；
- 解除历史归属或处理已有记录：Dialog 必须明确说明项目定义、历史时间段、历史归属和规则的处理结果。

# 10. Settings / Privacy 信息架构

## 10.1 页面职责

Settings / Privacy 负责：

- 说明本地存储和隐私边界；

- 管理导出目录；
- 导出加密备份；
- 导入并替换备份；
- 清空本地数据；
- 查看高级诊断。

## 10.2 页面结构

页面内分类：

- 常规；
- 采集；
- 隐私；
- 数据与备份；
- 高级。

这些是页面内分类，不增加一级导航。

## 10.3 高级诊断

普通区域只显示用户结论：

- 记录正常；
- 需要重启；
- 数据维护失败；
- 恢复尚未完成。

Collector、maintenance 和 recovery 原始字段仅在高级区域显示。

## 10.4 数据危险操作

导入并替换数据库、清空全部数据继续要求输入确认文字。

不得因 Timeline 使用两步确认而降低这些更高风险操作的确认等级。

## 10.5 刷新

不显示普通状态刷新按钮。

失败时可以提供重试、重新检查或恢复操作。

# 11. 通用状态与反馈

## 11.1 Loading

- 保留页面框架；
- 首次加载可使用少量 Skeleton；
- 自动刷新时保留旧数据；
- 不整页闪白。

## 11.2 Empty

只包括一句说明和最多一个合理操作。

例如：

```text
当日暂无时间记录
WorkTrace 开始记录后，时间段会显示在这里。
```

## 11.3 Error

- 使用用户可理解的说明；
- 有旧数据时保留旧数据；
- 可重试时显示重试；
- 不显示 traceback 或敏感路径。

## 11.4 Auto-saving

状态包括 dirty、saving、saved、failed 和 unknown。

要求保留用户输入，旧响应不覆盖新输入，unknown 不伪装成普通失败或成功。

## 11.5 Deleting

删除流程状态包括：

- confirm-step-1；
- confirm-step-2；
- deleting；
- failed；
- unknown；
- succeeded。

删除中只禁用同一目标的冲突操作，不禁用整个 Timeline，也不提前从 UI 移除目标。

## 11.6 Filter updating

Statistics 查询更新时：

- 保留旧结果；
- 显示“更新中…”；
- 禁用导出；
- 新结果接受后替换旧结果。

## 11.7 Blocking

隐私确认失败或恢复阻断属于 Blocking 状态。

必须说明记录未启动或仍被阻断，提供实际可执行的恢复路径，不显示无效继续按钮。

# 12. 可访问性基线

必须满足：

- 所有交互均可通过键盘到达；
- 统一 `:focus-visible`；
- 当前导航使用 `aria-current="page"`；
- Timeline 时间段列表 DOM 顺序与视觉倒序一致；
- Timeline 选中项具有选中语义；
- Drawer 和 Dialog 具有正确名称并正确管理焦点；
- 删除确认明确说明目标；
- 两步确认按钮不得使用含糊的“确定”；
- 删除取消后焦点返回原删除按钮；
- 高级操作关闭后焦点返回高级操作按钮；
- Overview “整理”入口具有明确的目标时间段和缺失字段语义；
- 自动保存不改变焦点；
- 每秒实时计时不触发屏幕阅读器播报；
- 自动保存不因每次键入频繁播报；
- 派生摘要具有可理解的辅助语义；
- 不能仅靠颜色区分用户描述和派生摘要；
- 图标按钮具有可访问名称和 Tooltip；
- 不能仅靠颜色表达进行中、错误、危险或选中；
- 150% 缩放下文字和控件不被裁切；
- Statistics 时间和项目筛选具有明确标签；
- Project Rules 项目名称、描述、上次使用和累计时间关系清晰。

# 13. 技术与架构边界

必须保持：

1. Overview 当前活动和 Recent 使用既有身份语义。
2. Overview 待整理只基于后端已返回的显式项目和描述事实，不新增持久化状态。
3. Overview 不建立第二套 Timeline 编辑、草稿或自动保存逻辑。
4. Timeline 只消费后端物化的时间段和活动明细。
5. UI 不自行推断归属和可见性。
6. UI 不自行重新归并活动。
7. LiveClock 继续是实时计时唯一来源。
8. Statistics 使用后端接受的统计事实。
9. Statistics 项目筛选在后端查询生效。
10. Statistics 查询和导出使用同一接受快照。
11. Project Rules 使用权威项目目录。
12. 项目累计时间和上次使用时间来自后端事实。
13. 自动保存复用既有 mutation owner。
14. 删除复用既有写入门和错误语义。
15. 删除结果区分 succeeded、failed 和 unknown。
16. 派生摘要不成为可写字段。
17. 所有资源保持本地。
18. 不增加浏览器存储。
19. 不增加前端构建链。
20. 不引入 React、Vue、Vite、Electron、Tauri 或本地 HTTP 服务。

## 13.1 允许的最小 DTO 调整

如果现有页面模型不足，可以增加最小字段。

描述展示：

```text
display_description
description_source = user | derived | none
```

Overview 待整理：

```text
needs_attention
missing_fields = project | description | project_and_description
```

这些字段可以由后端页面模型统一物化，也可以由前端基于同一时间段 DTO 中明确的 `project_id` 和原始 `description` 进行纯展示判断；不得写回数据库。

Project Rules 项目摘要：

```text
project_description
last_used_at
total_duration_seconds
```

删除能力：

```text
can_delete
delete_blocked_reason
```

Statistics 项目筛选：

```text
project_id = null | concrete_project | unclassified
```

不得采用：

- N+1 详情查询；
- 前端维护第二份活动缓存；
- 前端自行重算统计；
- 从 DOM 反向解析项目事实；
- 为自动保存建立新的全局状态容器；
- 为 Overview 建立独立编辑状态；
- 为两步删除建立复杂流程引擎。

# 15. 验收标准

## 15.1 Overview

1. 展示今日记录摘要条、当前活动、最近记录和待整理。
2. 不展示项目汇总。
3. 今日记录摘要条的统计口径来自完整权威投影，不从最近记录显示截断反向计算。
4. 不显示刷新按钮。
5. 当前活动主标题来自结构化 DTO 的 `resource_name`，不是项目名称、用户描述或派生摘要。
6. 当前活动持续时间使用 `current_live`；最近记录持续时间使用 `aggregate_live` 或 durable duration；两者允许不同。
7. 当前活动不解析组合 display 字符串，也不读取 current_session 作为展示内容。
8. 当前活动显式覆盖无快照、暂停、空闲、已排除和异常状态，不残留过期资源或敏感内容。
9. 当前活动仅在存在 current_session、稳定 projection 身份且 Timeline 可定位时可点击；否则真实禁用。
10. 待整理只包含今日已结束且缺少项目或用户描述的可见时间段。
11. 派生摘要不能使时间段退出待整理。
12. 当前进行中的时间段不进入待整理，但可以作为最近记录的第一条显示。
13. 待整理是最近记录的子集，不是互斥分区；待整理记录同时存在于最近记录。
14. 待整理最多显示三条，并提供剩余数量入口。
15. 点击整理进入 Timeline 对应时间段并聚焦缺失字段。
16. Overview 不直接编辑时间段。
17. 没有待整理项时显示轻量“今日记录已整理”，最近记录扩展使用空间。
18. 正常记录状态不重复显示。
19. 页面文案、空状态和 ARIA 全部使用“当前活动 / 最近记录 / 待整理”。

## 15.2 Timeline

15. 时间段按开始时间倒序排列。
16. 当前时间段位于最上方。
17. 不显示状态筛选。
18. 不显示 idle、error、excluded、suppressed 或已修正标签。
19. 只允许显示“进行中”运行事实。
20. 保留项目筛选。
21. 时间段列表显示描述或派生摘要。
22. 项目和时长位于同一行。
23. 描述独占一行。
24. 项目、时长和描述自动保存。
25. 不显示保存和取消按钮。
26. 删除时间段直接显示，不放入高级操作。
27. 高级操作只包括拆分、合并、复制和隐藏等低频操作。
28. 删除时间段使用单个 Dialog 内的两步确认。
29. 活动详情始终显示，不设置展开或收起。
30. 每条可删除活动直接显示删除活动入口。
31. 删除活动使用单个 Dialog 内的两步确认。
32. 删除按钮文案明确区分时间段和活动。
33. 删除 unknown 时不提前移除目标。
34. 活动删除后由后端模型更新时间段时长和派生摘要。
35. 自动保存、删除和实时刷新不破坏 mutation owner、revision、generation 和 LiveClock。

## 15.3 Statistics

36. 提供时间范围和项目筛选，两者可单独或组合使用。
37. 快捷按钮仅为今天、本周和本月。
38. 不提供最近 7 天。
39. 本周按本地周一至今天计算。
40. 不提供状态筛选或按状态视图。
41. 只保留按项目和按应用。
42. 筛选变化后自动查询。
43. 不显示应用筛选或加载按钮。
44. 查询期间保留旧结果并显示更新中。
45. 查询期间禁用导出。
46. 查询和导出使用同一接受快照。
47. 不在前端对聚合结果二次过滤。
48. 不显示单条描述或派生摘要。

## 15.4 Project Rules

49. 页面级只保留新建项目。
50. 每个项目直接显示新建规则、编辑和删除。
51. 不使用项目级更多菜单。
52. 项目折叠时显示项目名称。
53. 项目描述存在时显示项目描述，为空时省略描述行。
54. 显示上次使用时间和累计时间。
55. 上次使用和累计时间来自后端事实。
56. 新建和编辑项目共用 Drawer。
57. 新建规则从具体项目进入。
58. 新建规则 Drawer 不再选择目标项目。
59. 文件夹规则默认包含子文件夹，不显示包含子文件夹选项。
60. 项目搜索包含项目描述。
61. 项目排序支持最近使用、累计时间和项目名称。

## 15.5 Settings 和通用要求

62. Settings 将普通设置与高级诊断分层。
63. 自动刷新页面不显示普通刷新按钮。
64. 错误状态保留重试。
65. 派生摘要不写入描述、不触发自动保存、不冒充用户描述导出。
66. `1080×720` 和 `800×540` 均可完成核心操作。
67. 125% 和 150% 缩放下控件不被裁切。
68. 所有主要操作可通过键盘完成。
69. 未改变统计、导出、归属、隐私或数据库业务语义。

# 16. 阶段结论

阶段 2 最终目标确定为：

- 保留五个一级页面；
- 自动刷新页面不显示普通刷新按钮；
- Overview 保留今日记录摘要条、当前活动、待整理和最近记录；
- 当前活动展示原子级活动快照（`resource_name` 驱动），不展示项目时间段或派生摘要；
- 待整理是最近记录的已结束行动子集，不是与最近记录互斥的分区；
- 待整理只承载缺少项目或用户描述的已结束时间段，并导航到 Timeline 整理；
- 描述成为时间记录的核心扫描信息；
- 没有用户描述时显示最长三个不同活动标题构成的派生摘要；
- 派生摘要只用于展示；
- Timeline 使用倒序时间段列表，不暴露内部状态，只保留项目筛选；
- Timeline 项目和时长同一行，描述独占一行，并使用自动保存；
- Timeline 直接显示删除时间段，高级操作只承载低频操作；
- 时间段和活动删除均使用同一 Dialog 内两步确认；
- 活动详情始终显示；
- Statistics 使用时间与项目两个筛选维度并自动查询；
- Statistics 快捷范围为今天、本周和本月，只保留按项目和按应用；
- Project Rules 项目直接显示新建规则、编辑和删除；
- 项目折叠时显示描述、上次使用和累计时间；
- 新建和编辑项目共用 Drawer；
- 新建规则直接绑定项目；
- 文件夹规则默认包含全部子文件夹；
- Settings 保持页面内分类和高级诊断分层。

进入阶段 3 前，应先同步修改静态低保真原型，并重点验证：

- Overview 三个核心模块和摘要条的空间分配；
- 待整理与最近记录的子集关系是否在 UI 中清晰可读；
- 当前活动主标题是否真实反映当前资源而非项目；
- 当前活动与最近记录时长允许不同的可读性；
- Timeline 两层时间段列表的扫描效率；
- 描述和派生摘要的视觉区分；
- 项目和时长同一行在 150% 缩放下的可用性；
- 自动保存状态是否足够轻量；
- 删除时间段按钮是否容易找到但不过度突出；
- 两步删除确认是否安全且不过度冗长；
- 活动删除后时间段模型是否正确更新；
- Statistics 自动查询是否产生明显抖动；
- Statistics 查询和导出快照是否严格一致；
- Project Rules 三个直接操作是否在窄窗口中仍保持清晰；
- `800×540` Drawer 是否无需页面级横向滚动。
