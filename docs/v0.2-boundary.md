# WorkTrace v0.2 Boundary

WorkTrace v0.2 is the next product boundary after v0.1 Lite. It does not replace the current local-first architecture. It defines what may be added next without weakening WorkTrace's privacy and offline-first baseline.

## Product Positioning

WorkTrace remains a local Windows work-trace and timesheet helper.

All features that can reasonably run locally must remain free and usable without registration, login, payment, or network access. Paid features may only cover server-side AI enhancement.

v0.2 is defined as:

```text
local data security + encrypted migration + anonymous payment authorization + opt-in automatic AI suggestions
```

## Free Local Features

The following features must remain free and local:

- activity collection;
- local project management;
- local project rules, including file, folder, keyword, and exclusion rules;
- local project/session correction;
- manual notes;
- local statistics;
- Excel export;
- local data clearing;
- local database encryption;
- encrypted local export/import for device migration;
- privacy exclusion rules.

Paid status must not affect these features. Network failure, token exhaustion, subscription expiry, or server errors must only disable server-side AI enhancement.

## Paid Features

Paid features are limited to:

1. improving project classification accuracy through server-side AI;
2. generating editable project-session note drafts through server-side AI.

Paid features must not include local collection, local classification rules, local export, local encryption, local migration, or local data clearing.

## No Registration Requirement

WorkTrace must not require a WorkTrace account.

Payment should use anonymous authorization credentials such as a license key, token voucher, subscription code, or local entitlement credential. The client should store only the minimum authorization state needed to use paid AI features.

The client must not require or store the user's real name, email, phone number, payment account, card information, or other identity data unless a future feature separately and explicitly requires it.

## AI Consent Model

AI is off by default.

When the user enables AI, WorkTrace may automatically request AI suggestions within the user's selected data scope. It should not ask for approval before every request, but the initial consent and settings must clearly explain:

- AI may run in the background;
- AI may suggest project classifications;
- AI may draft project-session notes;
- data is sent to the server only within the selected scope;
- AI can be disabled at any time;
- AI results are marked as AI-generated or AI-suggested;
- user edits and manual classifications take priority;
- exclusion rules, hidden records, deleted records, and AI-disabled projects/apps/paths are never uploaded.

## AI Data Scope

v0.2 should support scoped AI permissions.

Minimum metadata scope may include activity time, duration, app name, process name, window title, detected resource type, resource display name, path hint, URI host, existing project names, local rule matches, and user correction history.

Clipboard text and user-written notes require separate permission. Clipboard upload must remain off by default.

v0.2 must not add file-body, email-body, webpage-body, chat-body, screenshot, OCR, keyboard, or mouse-content upload. Any future content-reading feature requires a separate version boundary and a separate consent model.

## AI Classification Rules

AI classification is an enhancement layer, not a replacement for local rules.

AI must not override:

- exclusion rules;
- user manual classifications;
- classifications the user has already accepted or edited;
- hidden or deleted records;
- projects, apps, or paths disabled for AI.

AI suggestions should be stored and displayed separately from local rule results and manual results. Once the user edits an AI suggestion, the result becomes manual and must not be overwritten by later AI runs.

## AI Session Notes

AI may generate project-session note drafts that are suitable for review and later use in law-firm time entries.

AI-generated notes are drafts. They must not automatically become final user notes unless the user accepts or edits them.

AI must not invent client names, matter names, legal conclusions, deliverables, calls, meetings, drafting work, review work, or other work content unless the authorized data provides a basis for it.

The UI should distinguish:

- AI draft;
- user-accepted AI note;
- user-edited AI note;
- fully manual note.

## AI Visibility, Audit, and Reversal

Users must be able to see which results came from AI.

v0.2 should keep a local encrypted audit record for AI operations, including feature type, time, data scope, model/prompt version where available, token usage, result status, and whether the user accepted, edited, rejected, or reverted the result.

Users should be able to clear AI drafts, reject AI suggestions, revert AI classifications, and disable AI for selected projects, apps, or paths.

## Server Boundary

The server is only for anonymous entitlement checks, token/subscription accounting, AI classification, and AI note drafting.

The server must not provide cloud sync, remote viewing, team management, employee monitoring, automatic time submission, or full database storage.

Server-side retention should be minimal. By default, the server should not retain raw activity records, raw window titles, raw paths, clipboard text, user notes, generated note text, project names, file names, client names, or matter names beyond what is strictly needed to process the request.

User data must not be used for model training unless the user separately opts in.

## v0.2 Non-Goals

v0.2 should not include:

- mandatory accounts;
- cloud sync;
- team management;
- employee monitoring;
- remote record viewing;
- automatic time submission;
- file-body, email-body, webpage-body, or chat-body reading;
- screenshots;
- screen recording;
- OCR;
- keyboard logging;
- mouse-content logging;
- automatic application updates;
- full frontend technology migration;
- Windows service, driver, or administrator-permission requirement.

Automatic updates and frontend migration should be handled as separate later-version decisions.

## Implementation Order

Recommended v0.2 order:

1. document and test the v0.2 boundary;
2. add local database encryption;
3. add encrypted export/import migration;
4. add local structures for AI settings, AI audit, AI drafts, and anonymous entitlement state;
5. add anonymous token/subscription authorization;
6. define and test AI payload filtering;
7. add AI classification suggestions;
8. add AI project-session note drafts;
9. add AI review, edit, reject, revert, and clear controls.

## Acceptance Criteria

v0.2 is acceptable only if:

- local free features work without registration, payment, or network access;
- paid status affects only server-side AI features;
- AI is off by default;
- AI can run automatically only after user opt-in;
- AI results are clearly marked;
- user edits and manual classifications are never overwritten by AI;
- excluded, hidden, deleted, and AI-disabled data is never uploaded;
- local data is encrypted;
- encrypted migration is supported;
- server failure cannot break local collection, correction, statistics, export, clearing, encryption, or migration.
