# WorkTrace v0.2 Phase 1C.0: First-Batch Field Encryption Scan

This document is the Phase 1C.0 scan and implementation design output. It is a research-only artifact: no business code, schema, or tests were modified to produce it. The goal is to systematically map every write, read, display, export, `.wtbackup`, and test path for the candidate first-batch encrypted fields, and to produce an executable Phase 1C implementation plan.

Phase 1C.0 does not implement field-level encryption. It only scans and designs.

## Scope And Boundaries

Phase 1C.0 respects the same v0.2 local security boundaries:

- no SQLCipher;
- no AI, server, payment, license, token, subscription;
- no new runtime dependencies;
- no collector/UI/export behavior changes;
- no database schema changes;
- no blind index implementation (that is Phase 1D).

The scan covers the codebase as of commit `f274e13` (`fix: harden encrypted backup import safety`).

## 1. Scanning Summary

The scan examined:

- `worktrace/schema.sql` — column definitions, constraints, indexes
- `worktrace/collector/` — `collector.py`, `state_machine.py`, `auto_activity_recorder.py`, `heartbeat.py`
- `worktrace/services/` — `activity_service.py`, `clipboard_service.py`, `resource_service.py`, `session_note_service.py`, `recovery_service.py`, `privacy_service.py`, `project_inference_service.py`, `folder_rule_service.py`, `context_service.py`, `timeline_service.py`, `statistics_service.py`, `export_service.py`, `secure_backup_service.py`, `settings_service.py`
- `worktrace/resources/` — `resource_builders.py`, `resource_policy.py`, detectors
- `worktrace/exports/` — `excel_exporter.py`
- `worktrace/api/` — `dto.py`, `timeline_api.py`, `backup_api.py`
- `worktrace/ui/` — `timeline_view.py`, `overview_view.py`, `statistics_view.py`, `settings_view.py` (legacy package, deleted in Phase 6F)
- `worktrace/security/` — `crypto.py`, `kdf.py`, `key_manager.py`, `backup_format.py`
- `worktrace/formatters.py`
- `tests/` — all test files

Key findings:

1. **Several preferred first-batch fields cannot be encrypted in Phase 1C** because they are used in SQL `WHERE` equality clauses (`clipboard_service.find_activity_for_clipboard_event`). These require blind indexes and are deferred to Phase 1D.
2. **Several disputed fields can be included in Phase 1C** because they are only read via service-layer functions (not SQL `WHERE`), so transparent decrypt in the service layer preserves all matching/inference behavior.
3. **`activity_clipboard_event.text_hash` should not be encrypted** — it is already a SHA-256 hash and is used in a SQL `WHERE` clause for duplicate detection.
4. **The Phase 1A crypto foundation is directly reusable** for field encryption: `encrypt_aead`/`decrypt_aead` with the `wtenc1:` envelope, AAD binding, and `is_encrypted_envelope` detection.
5. **The exclusion/privacy path is not affected** by field encryption because exclusion decisions are made on in-memory `ActiveWindow`/`DetectedResource` objects before any DB write.

### Write path observations

1. **Single primary write function per table**: `activity_service.create_activity` (activity_log), `clipboard_service.record_clipboard_event` (activity_clipboard_event), `resource_service.create_or_update_activity_resource` (activity_resource), `session_note_service.set_session_note` (project_session_note). This simplifies adding encryption at the service boundary.

2. **`activity_log.note` has multiple UPDATE paths**: `update_activity_note`, `update_activity_fields`, `recovery_service.mark_record_error`. All three must encrypt on write.

3. **`activity_resource` has an UPDATE path in `_sync_activity_resource_after_path_update`** that updates `path_hint`, `display_name`, `identity_key`, `path_key`, `resource_kind`, `resource_subtype` — but NOT `window_title`, `metadata_json`, `app_name`, `process_name`. The UPDATE path must encrypt `path_hint` and `display_name`.

4. **Backup import (`_insert_rows`)** is generic: it uses `PRAGMA table_info` to discover columns and INSERTs raw values. If fields are encrypted at rest, import must re-encrypt plaintext backup payload values before INSERT.

5. **Recovery service is a hidden writer**: `_recover_cross_midnight_row` calls `create_activity` (covered by service-layer encryption), but `mark_record_error` does a raw UPDATE on `note` — this needs a separate encryption call.

6. **No UI direct SQL**: all UI writes go through `worktrace.api` → `worktrace.services`.

6. **`export_service.export_all_local_data`** does a raw `SELECT *` table dump. If fields are encrypted at rest, this function must decrypt all encrypted columns before writing to the export sheet, or the export will contain ciphertext.

## 5. Encryption Boundary Recommendations

### 5.1 New module: `worktrace/security/field_crypto.py`

This module is the only place that imports both `worktrace.security.crypto` and `worktrace.security.key_manager`. It never imports `worktrace.db`. Service layers call `field_crypto` functions and handle DB I/O.

Proposed API:

```python
# worktrace/security/field_crypto.py

from __future__ import annotations

from ..security.crypto import (
    CryptoError,
    decrypt_aead,
    encrypt_aead,
    is_encrypted_envelope,
)
from ..security.key_manager import create_or_load_local_key


class FieldCryptoError(Exception):
    """Generic error for field encryption/decryption failures."""


def _build_aad(table: str, column: str, row_id: int | str) -> bytes:
    """Build AAD binding the ciphertext to its row/column location."""
    return f"WorkTrace|db-field|v1|{table}|{column}|{row_id}".encode("utf-8")


def _get_key() -> bytes:
    """Load the local data key from the Phase 1A keyring."""
    return create_or_load_local_key().key


def encrypt_field(
    table: str, column: str, row_id: int | str, value: str | None
) -> str | None:
    """Encrypt a field value. Returns None for None. Idempotent: if the
    value is already a wtenc1: envelope, it is returned unchanged."""
    if value is None or value == "":
        return value
    if is_encrypted_envelope(value):
        return value
    key = _get_key()
    aad = _build_aad(table, column, row_id)
    try:
        return encrypt_aead(value.encode("utf-8"), aad, key)
    except CryptoError as exc:
        raise FieldCryptoError("field encryption failed") from exc


def decrypt_field(
    table: str, column: str, row_id: int | str, value: str | None
) -> str | None:
    """Decrypt a field value. Returns None for None. If the value is not
    an encrypted envelope (legacy plaintext), it is returned unchanged."""
    if value is None or value == "":
        return value
    if not is_encrypted_envelope(value):
        return value  # legacy plaintext passthrough
    key = _get_key()
    aad = _build_aad(table, column, row_id)
    try:
        return decrypt_aead(value, aad, key).decode("utf-8")
    except CryptoError as exc:
        raise FieldCryptoError("field decryption failed") from exc


def encrypt_row_fields(
    table: str, row: dict, row_id: int | str, encrypted_columns: tuple[str, ...]
) -> dict:
    """Return a copy of ``row`` with the specified columns encrypted."""
    out = dict(row)
    for col in encrypted_columns:
        if col in out:
            out[col] = encrypt_field(table, col, row_id, out[col])
    return out


def decrypt_row_fields(
    table: str, row: dict, row_id: int | str, encrypted_columns: tuple[str, ...]
) -> dict:
    """Return a copy of ``row`` with the specified columns decrypted."""
    out = dict(row)
    for col in encrypted_columns:
        if col in out:
            out[col] = decrypt_field(table, col, row_id, out[col])
    return out
```

### 5.2 AAD convention

AAD binds ciphertext to its row/column location to prevent ciphertext relocation attacks:

```
WorkTrace|db-field|v1|<table>|<column>|<row-id>
```

- `<table>`: the SQLite table name (e.g. `activity_log`)
- `<column>`: the column name (e.g. `note`)
- `<row-id>`: the integer primary key (`id` column)

The AAD is reconstructable at decrypt time from non-encrypted data (table name, column name, row primary key), so no extra DB read is needed.

### 5.4 Idempotent encryption

`encrypt_field` checks `is_encrypted_envelope(value)` before encrypting. This means:
- re-running migration on already-encrypted rows is safe
- re-calling `encrypt_field` on a value that was already encrypted does not double-encrypt

### 5.5 Legacy plaintext compatibility

`decrypt_field` checks `is_encrypted_envelope(value)` before attempting decryption. If the value is not an envelope (legacy plaintext), it is returned unchanged. This enables:
- incremental migration (mixed plaintext/ciphertext rows)
- rollback (if migration is rolled back, the DB still has plaintext and reads work)

### 5.6 Decryption failure handling

`decrypt_field` raises `FieldCryptoError` on decryption failure. It does NOT:
- silently return the ciphertext
- log the ciphertext, key, or plaintext
- fall back to plaintext

The caller (service layer) should catch `FieldCryptoError` and surface a generic error to the user. The service layer should not continue with corrupted data.

### 5.7 Logging hygiene

`field_crypto.py` logs only:
- operation name (encrypt/decrypt)
- result (success/failure)
- exception type

It never logs:
- the plaintext value
- the ciphertext envelope
- the AAD
- the key
- the table/column/row-id (these are in the AAD, not in log messages)

### 5.8 Boundary rules

- `field_crypto.py` lives in `worktrace/security/`
- It imports only from `worktrace.security.crypto` and `worktrace.security.key_manager`
- It never imports `worktrace.db`, `worktrace.services`, or `worktrace.collector` (the legacy `worktrace.ui` package was deleted in Phase 6F)
- Service layers (`activity_service`, `clipboard_service`, `resource_service`, `session_note_service`) call `field_crypto` functions
- The UI never imports `field_crypto` directly — it goes through `worktrace.api`
- `secure_backup_service` calls `field_crypto` to re-encrypt on import

| `field_encryption_schema_version` | `"0"` | `0` = no encryption, `1` = Phase 1C encryption applied |
| `field_encryption_migrated_at` | `""` | ISO 8601 timestamp of the last successful migration |

Both keys are added to `seed_defaults` and to `NON_MIGRATABLE_SETTINGS` in `secure_backup_service` (they are machine-local state, not backup content).

### 6.2 Migration trigger

**Recommended: service/test-only migration in Phase 1C, no auto-migration at startup.**

Rationale:
- Auto-migration at startup risks blocking app launch if the keyring is corrupted or the DB is large.
- A Settings/Privacy prompt adds UI complexity that Phase 1C should avoid.
- A service-layer migration function (`field_crypto.migrate_phase_1c()`) can be called:
  - manually from tests
  - from a future Settings/Privacy button in Phase 1C.1 or later
  - lazily on first read/write of an encrypted column (opportunistic per-row migration)

**Opportunistic per-row migration** is the simplest approach:
- On write: `encrypt_field` is called, which encrypts the value. The row is now encrypted.
- On read: `decrypt_field` is called, which handles both plaintext and ciphertext. No migration needed.
- Background migration: a one-shot function `migrate_phase_1c()` iterates all rows in the encrypted tables and calls `encrypt_field` on each encrypted column. This is idempotent and can be run once.

### 6.3 Migration flow

```
migrate_phase_1c():
  1. Check field_encryption_schema_version. If already "1", return.
  2. Enter secure_import_guard (reuse Phase 1B.1 guard to pause collector).
  3. Begin transaction.
  4. For each table in PHASE_1C_ENCRYPTED_COLUMNS:
       For each row in table:
         For each encrypted column:
           If not is_encrypted_envelope(value):
             encrypt_field(table, column, row_id, value)
             UPDATE table SET column = ? WHERE id = ?
  5. Set field_encryption_schema_version = "1".
  6. Set field_encryption_migrated_at = now_str().
  7. Commit transaction.
  8. Exit guard (leaves app paused, user must resume manually).
  On failure:
    - Rollback transaction.
    - Restore prior guard state (guard handles this).
    - field_encryption_schema_version remains "0".
```

### 6.4 `.wtbackup` interaction

**Export**: the backup payload should contain **business plaintext**, not field-level ciphertext. Rationale:
- The backup is already protected by the `.wtbackup` envelope encryption.
- The backup may be imported on a different machine with a different local key, so field-level ciphertext would be undecryptable.
- `secure_backup_service._build_export_payload` should **decrypt** encrypted columns before serializing to the backup payload.

**Import**: after writing the plaintext backup payload to the DB, `secure_backup_service._replace_import` should **re-encrypt** the Phase 1C encrypted columns using the local key. This happens inside the secure import guard.

Flow:
```
import_encrypted_backup():
  1. Enter secure_import_guard.
  2. Decrypt .wtbackup envelope → plaintext payload.
  3. _replace_import: INSERT plaintext rows into DB.
  4. Re-encrypt Phase 1C columns: for each row in encrypted tables,
     encrypt_field(table, column, row_id, value).
  5. Set field_encryption_schema_version = "1".
  6. Exit guard.
```

### 6.5 Old plaintext DB

- Before migration: `decrypt_field` returns plaintext unchanged (legacy passthrough). The UI displays correctly.
- After migration: `decrypt_field` decrypts `wtenc1:` envelopes. The DB file no longer contains plaintext for Phase 1C fields.
- After rollback: the transaction rolls back, so the DB still has plaintext. `field_encryption_schema_version` remains `"0"`. Reads work via legacy passthrough.

## 9. Key Risks

1. **`export_service.export_all_local_data` raw `SELECT *`**: this function dumps every column directly. If encryption is at rest and this function is not updated, the export will contain ciphertext. It must be updated to decrypt all encrypted columns.

2. **`recovery_service.mark_record_error` raw UPDATE**: this function appends error text to `note` via raw SQL (`UPDATE activity_log SET note = COALESCE(note || CHAR(10), '') || ?`). If `note` is encrypted at rest, this concatenation produces garbage. The function must be updated to read-decrypt-append-encrypt.

3. **`activity_service.update_activity_fields` dynamic SQL**: this function builds a dynamic UPDATE statement. If `note` is in the fields dict, it must be encrypted before being passed to the SQL.

4. **Performance**: transparent decrypt on every read adds overhead. For `get_activities_by_date` (which can return hundreds of rows), decrypting `note` for each row may be noticeable. Consider lazy decryption (decrypt only when the field is actually displayed/exported, not when the row is loaded).

5. **Key loss**: if the DPAPI keyring is corrupted or the Windows user profile is deleted, encrypted fields become undecryptable. The `.wtbackup` export/import path remains the recovery mechanism (backup payload is plaintext).

6. **Mixed plaintext/ciphertext during migration**: if the app crashes during migration, some rows may be encrypted and others not. The `decrypt_field` legacy passthrough handles this gracefully — encrypted rows decrypt, plaintext rows pass through. A re-run of `migrate_phase_1c()` completes the migration.

7. **`activity_resource.path_key` remains plaintext**: `path_key` is a normalized path derived from `path_hint`. It has an index (`idx_activity_resource_path`). While `path_hint` is encrypted, `path_key` remains plaintext, which means the normalized path is still visible in the DB. This is a known limitation — `path_key` is explicitly excluded from Phase 1C and will be addressed in Phase 1D with blind indexes.
