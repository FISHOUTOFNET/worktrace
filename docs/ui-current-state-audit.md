# 二、执行结论

## 1. 可以开始 UI 重构

WorkTrace 当前已经具备启动 UI 重构的条件：

- 五个主要页面均已具备完整或基本完整的功能表面；
- WebView 是唯一正式 UI，不再存在 Tkinter 双轨迁移问题；
- 前端与后端之间具有固定 bridge 边界；
- LiveClock、页面请求、选中状态和写操作并发具有明确所有者；
- 统计、导出、规则、备份和隐私操作均有契约测试；
- 最近一次明确记录的完整 Standard CI 中，Python 全量测试、WebView Node 行为测试和 Windows 打包验证均通过。

但本轮应被定义为：

> 在不改变产品语义和运行时契约的前提下，重构信息架构、视觉系统和桌面交互。

不应把 UI 重构扩大为：

- 更换 Python 后端架构；
- 改写 LiveClock；
- 重建状态机；
- 引入本地 HTTP 服务；
- 改用 Electron、Tauri；
- 立即引入 React、Vue 或大型前端工具链。

当前无构建步骤的原生 HTML、CSS、JavaScript 架构是有意选择：它避免 Node 构建链、额外打包路径和本地网络服务，保持 PyInstaller 分发简单。以现有产品规模判断，它足以支持本轮 UI 重构。

## 2. 当前 UI 的主要问题不是“功能缺失”

当前主要问题是：

- 缺少统一设计系统；
- 页面信息层级不够清晰；
- 相同控件存在多套样式；
- 桌面端空间利用不合理；
- Timeline 仍是列表式管理页，而非真正清晰的时间工作台；
- Settings 暴露过多运行时技术状态；
- 键盘操作、焦点样式和动态状态播报不足；
- 自动化测试保护了业务契约，但未充分保护真实视觉和桌面交互。

因此不需要重新设计产品功能，只需要重新组织和呈现现有功能。

# 三、当前前端架构

## 1. 技术结构

当前前端为单页 WebView 应用：

```text
index.html
    ↓
本地 styles.css
    ↓
经典 JavaScript 脚本，按固定顺序加载
    ↓
window.WorkTraceApp 单一命名空间
    ↓
固定 pywebview bridge
    ↓
worktrace.api / ApplicationServices
```

所有 HTML、CSS 和 JavaScript 都是本地资源；前端不使用 CDN、远程字体、浏览器存储或本地 HTTP 服务。bridge 是唯一数据路径。

当前脚本被拆分为：

- `core.js`；
- `overview.js`；
- `timeline_request_state.js`；
- `timeline.js`；
- `statistics.js`；
- `settings.js`；
- 六个 Project Rules 模块；
- `init.js`。

这些脚本通过固定顺序加载，不使用 ES Module、动态模块寻址或 bundler。

## 2. 当前运行时边界是合理的

以下逻辑已经形成清晰边界，应当保留：

- `init.js` 独占 pywebview 访问和应用初始化；
- bridge 方法是固定白名单，不允许页面动态调用任意后端方法；
- LiveClock v2 有唯一运行时 store；
- 每秒本地 ticker 只更新 DOM，不重新调用后端；
- 页面结构变化才触发页面数据刷新；
- Timeline 有独立的请求、选中、写操作和数据世代 owner；
- 数据库替换后统一清理前端缓存和请求状态。

前端测试也明确禁止：

- 第二套 LiveClock；
- 页面自行判断活动是否应物化；
- ticker 读取结构缓存；
- 页面通过网络或浏览器存储建立旁路。

这些不是 UI 重构障碍，而是应当保留的安全边界。

# 四、五个页面的现状

## 1. Overview

### 当前内容

Overview 目前包含：

- 刷新按钮；
- 今日日期；
- 今日总时长；
- 今日项目数；
- 已归类时长；
- 未归类时长；
- 当前活动；
- 最近活动；
- 错误提示。

### 当前优点

- 当前活动和 KPI 已接入统一 LiveClock；
- Recent 的实时行使用后端提供的精确时钟；
- 时间数字已使用等宽数字；
- 动态文字经过转义处理。

### 当前问题

1. “今日日期”与核心时长数据使用相同 KPI 卡片权重，浪费重要首屏空间。
2. 五张 KPI 卡片缺乏主次关系。
3. 当前活动仅作为普通文本置于卡片中，没有成为页面视觉中心。
4. 没有项目时间分布或未归类处理入口。
5. Overview 有错误状态，但没有独立、明确的加载状态。
6. 当前活动和 Recent 各自占一整张卡片，页面容易形成“卡片堆叠”。

### 建议定位

Overview 应只负责：

- 显示当前活动；
- 显示今日核心时间；
- 显示主要项目分布；
- 提醒未归类或异常记录。

它不应成为完整统计页。

## 2. Timeline

### 当前内容

Timeline 目前包含：

- 前一天、日期选择、后一天、今日；
- 当日总时长；
- 当前活动；
- 左侧时段列表；
- 右侧活动耗时详情；
- 删除时段；
- 合并到前一时段；
- 合并到后一时段；
- 拆分；
- 复制；
- 修改项目；
- 修改申报时长；
- 修改备注；
- 加载、空数据和错误状态。

### 当前优点

Timeline 是当前前端中契约最严格的页面：

- 精确匹配 LiveClock；
- 防止过期页面响应覆盖当前页面；
- 防止过期选中详情覆盖新选中项；
- 写操作有 pending、unknown、confirmed success、confirmed failure 状态；
- 写后重新读取页面模型；
- 编辑中避免实时刷新覆盖用户输入。

### 当前问题

1. 当前所谓 Timeline 实际是“时段列表 + 详情列表”，缺少按一天时间位置展示的真正时间轴。
2. 左右两个通用白色卡片权重相同，用户难以快速判断主工作区。
3. 删除、前合并、后合并、拆分、复制五个结构操作始终同时出现，操作密度过高。
4. 详情、结构操作和字段编辑全部堆在右侧面板内，纵向负担较重。
5. 选中项是可点击的 `<div>`，缺乏原生键盘可操作性。
6. 小窗口下只是将左右区域改成上下排列，详情和编辑会被推到很长的页面下方。
7. 当前活动、当日总时长和日期导航分散在多个区域，缺乏统一的工作台标题栏。
8. 页面没有项目和状态筛选器。

### 建议定位

Timeline 应成为首要重构页面，并形成：

- 固定日期和筛选工具栏；
- 真正按时间排序和定位的主体区域；
- 选中状态清晰的活动记录；
- 右侧 Inspector；
- 次级菜单承载低频结构操作；
- 小窗口下覆盖式详情 Drawer。

## 3. Statistics / Export

### 当前内容

当前页面包含：

- 开始和结束日期；
- 今天、最近七天、本月快捷范围；
- 加载按钮；
- 总时长、活动数、项目数、应用数；
- 按项目、应用、状态的三张表；
- 导出预览；
- CSV 导出；
- 已完成活动范围提示。

### 当前优点

- 日期范围限制清晰；
- 读取和导出共享已接受的统计 revision；
- 日期改变后旧导出凭证会失效；
- 导出有进行中、成功、取消和失败状态；
- 动态表格值经过转义；
- CSV 是唯一公开导出能力，隐私边界明确。

### 当前问题

1. 四张概览卡片和三张统计表缺乏明显主次。
2. 在应用默认 1080px 宽度下，三列表格会触发 `max-width: 1100px` 规则并全部垂直堆叠，页面很长。
3. “统计仅包含已完成活动”的相同提示出现两次。
4. 导出预览占据一张完整大卡片，但导出属于次级操作。
5. “应用数”和“按应用统计”的实际价值可能低于项目统计，却获得同等视觉权重。
6. 表格没有统一的排序、固定列或密度规则。
7. 缺少按日趋势；但不应因此引入复杂图表系统。

### 建议定位

统计页应以项目统计表为主，趋势图为辅，CSV 导出收纳到右上角或独立次级区域。

## 4. Project Rules

### 当前内容

Project Rules 已经具备：

- 新建规则；
- 新建项目；
- 排序；
- 项目卡片；
- 项目内规则列表；
- 编辑项目；
- 新增规则；
- 删除项目；
- 删除规则；
- 文件夹和关键词规则；
- 创建时应用到历史记录；
- 右侧创建或编辑面板。

### 当前优点

- 已经采用右侧 Drawer，方向合理；
- 旧的大量内嵌表单和批量工具已被移除；
- 新建项目和新建规则共用面板；
- 规则创建后的历史回填有明确结果；
- 项目目录与 Timeline 共享同一前端缓存协调逻辑。

### 当前问题

1. 当前采用“一个项目一张卡片，内部再嵌套规则行”，项目增多后扫描效率较差。
2. 每张项目卡片都重复显示编辑、增加规则和删除三个按钮。
3. 缺少搜索、筛选和紧凑的规则总览。
4. 项目删除使用原生 `window.confirm`，规则删除使用自定义 Modal，交互不一致。
5. 项目删除成功后调用错误 Banner 显示“项目已删除”，成功和错误语义混用。
6. Drawer 虽然有 dialog 标记和关闭按钮，但未发现统一的 Escape、焦点锁定和关闭后焦点恢复机制。
7. 文件夹规则和关键词规则在视觉上仍依赖大量页面专用按钮类。

### 建议定位

短期不必立即改成复杂数据表。可先使用：

- 项目紧凑列表；
- 展开后显示规则；
- 固定的主要操作；
- 统一 Drawer；
- 统一确认 Dialog；
- 搜索和排序。

数据规模确认较大后，再决定是否完全表格化。

## 5. Settings / Privacy

### 当前内容

Settings 当前包含：

- 本地存储说明；
- 剪贴板记录开关；
- 导出目录状态；
- 数据库维护状态；
- 恢复状态；
- 阻断原因；
- Collector 状态；
- 用户暂停状态；
- 隐私说明；
- 加密备份导出；
- 备份清单预览；
- 加密备份导入；
- 清空本地数据。

### 当前优点

- 敏感输入不会保存在全局前端状态；
- 操作完成后清除口令；
- 导入和清空均要求明确确认文字；
- 多个设置写操作之间具有统一互斥保护；
- 设置页主要使用 `textContent`，避免不安全动态 HTML。

### 当前问题

1. 用户设置、隐私说明、运行时诊断、备份和危险操作全部出现在同一个卡片网格中。
2. “维护进行中”“维护恢复完成”“恢复阻断”“Collector 状态”等内部技术信息直接暴露给普通用户。
3. 技术状态没有转化为用户可理解的结论，例如“记录正常”“需要重启”“数据维护失败”。
4. 备份导出、清单预览、数据替换和清空数据都在同一视觉层级。
5. 危险操作与普通设置虽然颜色不同，但信息架构上的隔离不足。
6. 当前自适应网格会根据宽度自动排列卡片，阅读顺序容易随窗口尺寸变化。

### 建议定位

拆分为：

- General；
- Collection；
- Privacy；
- Data and Backup；
- Advanced / Diagnostics。

普通用户只应看到可执行结论；维护、恢复和 Collector 原始状态放入默认折叠的 Advanced。

# 六、主要视觉问题

## 必须解决

### 1. 未形成显式设计 token

当前样式主要通过大量硬编码颜色、字号、边框、圆角和间距构成。例如按钮、输入框、卡片、错误提示、成功提示和页面背景在不同页面分别声明。

应建立最小 token：

- 背景；
- surface；
- 文字三级；
- 边框；
- accent；
- success、warning、danger；
- 字号；
- 间距；
- 圆角；
- 控件高度；
- focus ring。

### 2. 单一 stylesheet 已明显膨胀

`styles.css` 按页面持续追加，新旧 Project Rules 样式甚至在文件后部再次覆盖前面的相同选择器。

这不会立即造成业务错误，但会导致：

- AI 难以判断真正生效的规则；
- 后续页面容易继续复制样式；
- 修改一个控件时遗漏其他同类控件；
- CSS 优先级越来越依赖文件顺序。

### 3. 卡片使用过多

Overview、Timeline、Statistics、Rules 和 Settings 都大量使用白色圆角卡片。卡片没有真正表达不同层级，而是成为默认容器。

应减少卡片数量，更多使用：

- 页面区块；
- 分隔线；
- 表格；
- Inspector；
- 工具栏；
- 单一主 surface。

### 4. 字体和数字规范不统一

当前默认字体顺序以 Microsoft YaHei 在 Segoe UI 之前，不利于形成 Windows 11 原生感。部分时间使用等宽数字，其他统计数字未统一处理。

# 七、主要交互问题

## 必须解决

### 1. 应用最小窗口适配不足

实际窗口默认尺寸是 `1080×720`，最小尺寸是 `800×540`，而不是此前设想的以 1280px 为最低基准。

当前左侧导航固定 220px，主内容还有左右各 32px padding。在 800px 宽度下，实际内容区会明显受限。现有响应式规则主要只把页面内部两列改为一列，没有处理侧边栏压缩或收起。

第一轮 UI 设计必须以以下尺寸为硬约束：

- 默认：1080×720；
- 最小：800×540；
- 补充：1366×768、1920×1080；
- Windows 125% 和 150% 缩放。

### 2. 键盘和焦点支持不足

当前导航按钮可以获得浏览器默认焦点，但没有统一 `focus-visible` 系统；Timeline 选中项是点击式 `<div>`；未发现通用键盘导航、Escape 关闭 Drawer、焦点锁定或焦点恢复逻辑。

应当增加：

- 全局 `:focus-visible`；
- 当前导航 `aria-current`；
- 可点击记录使用 button 或正确的键盘语义；
- Drawer 和 Modal 的 Escape、初始焦点、焦点锁定、关闭后恢复；
- 动态错误和成功状态使用 `aria-live` 或 `role=status`；
- 图标按钮必须有可访问名称。

### 3. 状态反馈没有统一语义

当前存在：

- `error-banner`；
- `edit-status-*`；
- `stats-export-status.*`；
- `rules-panel-status.*`；
- `settings-error`；
- 多种页面专用 loading。

应建立统一的：

- Inline Alert；
- Toast；
- Field Error；
- Loading State；
- Empty State；
- Blocking Error。

### 4. 删除和危险操作不一致

原生 `window.confirm`、自定义 Modal、输入确认文字和直接按钮同时存在。它们应根据风险分级统一：

- 普通删除：Confirm Dialog；
- 影响历史数据：带说明和可选项的 Dialog；
- 替换数据库、清空数据：确认文字；
- 不得使用浏览器原生 `window.confirm`。

# 八、测试保护面

## 当前已有的保护

自动化测试已经较好地保护：

- 资源加载顺序；
- bridge 固定边界；
- 无网络、无浏览器存储；
- LiveClock 单一所有者；
- 页面关键 DOM ID；
- 统计导出 revision；
- Settings 敏感数据；
- Rules 页面功能入口；
- Windows exe 和 installer 打包。

## 当前缺口

现有大量前端契约测试是直接读取 HTML、CSS 和 JavaScript 文本完成的，并不启动真实 GUI。

因此它们不能充分发现：

- 实际文字溢出；
- Windows 字体差异；
- 125% 或 150% 缩放错位；
- Drawer 超出可视区；
- 焦点不可见；
- 键盘无法选择 Timeline；
- 颜色对比度不足；
- 页面滚动和固定区域冲突；
- 重构后的视觉回退。

项目已经有 Windows 手工 UI 验收清单，但该清单主要检查功能可用性、刷新和窗口稳定性，没有完整的尺寸、缩放和可访问性矩阵。

UI 重构中应增加少量稳定测试：

- DOM 行为测试；
- 键盘导航测试；
- Drawer 开关和焦点测试；
- 1080×720 与 800×540 截图验收；
- 125% 和 150% Windows 手工缩放验收。

不建议建立大量逐像素快照测试。

# 九、必须保持的产品行为

UI 重构不得改变以下现有契约：

1. Overview 拥有当前活动的主要呈现。
2. Timeline 只能将一个持久化开放活动匹配为一个实时条目。
3. 前端不得在多个候选 LiveClock 中自行选择。
4. 无效 LiveClock 停止计时并保留持久化时长。
5. Recent 第一条和当前活动保持统一运行时身份。
6. Timeline、Statistics 和 Export 使用后端事实，而不是前端重新计算。
7. 开放活动只允许现有契约允许的字段编辑。
8. 数据替换后必须清除全部前端缓存和过期请求。
9. Project Rules 和 Timeline 不得建立第二份项目目录缓存。
10. 隐私确认前不得开始敏感采集。
11. 所有前端资源保持本地，不增加网络依赖。
12. 前端不得直接访问数据库或后端 service。

# 十一、重构风险

## 高风险

1. 改动 LiveClock 对应的 `data-*` 属性，导致计时停止或身份串行。
2. 重建 Timeline DOM 时破坏选中 owner 和编辑保护。
3. 把页面状态复制到第二个前端 store。
4. 修改统计展示时重新用前端活动数据计算总时长。
5. 为 UI 方便改变 bridge DTO。
6. 删除当前测试依赖的 DOM ID，却未同步建立更稳定的测试锚点。

## 中风险

1. 拆分 CSS 后加载顺序变化。
2. Drawer 重构后事件重复绑定。
3. 统一按钮时弱化危险操作。
4. 小窗口采用 overlay 后出现滚动和焦点问题。
5. 每秒重建 Timeline 或 Overview 大片 DOM，引入性能退化。

## 低风险

1. 字体、颜色和间距 token；
2. 导航图标和文字层级；
3. 卡片减少；
4. 状态组件统一；
5. 表格密度优化；
6. 页面标题栏整理。

# 十五、最终判断

WorkTrace 当前符合“基本功能已经实现，可以开始 UI 重构”的条件。

当前前端架构不漂亮，但并不混乱到需要推倒重做。其运行时、隐私、桥接、请求并发和数据一致性边界已经相当严格。真正需要重构的是：

- 页面布局；
- 信息层级；
- 视觉 token；
- 基础控件；
- 桌面窗口适配；
- 键盘和焦点；
- 状态反馈；
- 测试中的真实视觉验收。

最佳策略不是更换技术栈，而是：

> 保留现有运行时和页面业务逻辑，在同一原生 WebView 技术栈内，先建立最小设计系统，再以 Timeline 为代表页面逐步替换表现层。

阶段 1 已完成，可以进入阶段 2：信息架构与低保真设计。
